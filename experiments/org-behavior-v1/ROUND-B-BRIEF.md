# B组任务书：Issue #71 — Bot API 支持查询群列表和成员详情

> 组织模式：**全连接脉冲型（Pulsed Full-Connect）**
> 任何龙虾可直接 @任何龙虾，阶段间有强制全员同步脉冲。

---

## 一、沟通规则

### 角色

| 角色 | 职责 |
|------|------|
| **Sensor/Designer** | 理解需求、定义用户场景、设计API和权限方案 |
| **Builder** | 编写技术方案、核心逻辑伪代码、确认可行性 |
| **Reviewer** | 审查方案一致性、安全性、与现有架构的兼容性 |

> 注意：本轮 **没有 PM 角色**。A组的 PM 龙虾在本轮担任观察员，不参与执行。

### 信息流

```
Sensor/Designer ←→ Builder ←→ Reviewer
      ↕                          ↕
      └──────────────────────────┘
           任何人可 @任何人
         阶段间有同步脉冲
```

### 规则（必须严格遵守）

**规则1 — 自由沟通**：任何龙虾可以在任何时候 @任何其他龙虾沟通。没有信息路由限制。

**规则2 — 阶段划分**：任务分四个阶段，按顺序推进：

| 阶段 | 主导者 | 产出 |
|------|--------|------|
| 1. 需求理解 | Sensor/Designer | 用户故事 + 场景定义 |
| 2. 方案设计 | Sensor/Designer | API spec + 数据模型 + 权限方案 |
| 3. 技术实现 | Builder | 实现方案 + 核心伪代码 |
| 4. 审查 | Reviewer | review 意见 + 修订建议 |

**规则3 — 同步脉冲**：每个阶段完成时，**全员必须各发一条同步消息**，确认理解一致。

格式：
```
[同步X] 阶段X完成确认
- 我的理解：...
- 同意的部分：...
- 有疑虑的部分：...
- 建议：...
```

**全员确认无分歧后，才能进入下一阶段。** 有分歧当场讨论解决。

**规则4 — 阶段内自由参与**：虽然每个阶段有主导者，但其他龙虾可以随时插嘴：
- 阶段1：Builder 可以提前指出技术约束，Reviewer 可以质疑场景合理性
- 阶段2：Builder 可以直接说"这样设计后端改动太大"，Reviewer 可以提前指出安全风险
- 阶段3：Designer 可以随时被 @确认业务细节，Reviewer 可以提前 review 关键决策
- 阶段4：Reviewer 有问题直接 @Builder 和 Designer 讨论

## 二、任务需求

### 2.1 背景

当前 Bot API 已有基础的群组查询能力：
- `GET /v1/bot/groups` — Bot 获取所在群组列表
- `GET /v1/bot/groups/:group_no` — Bot 获取群组信息
- `GET /v1/bot/groups/:group_no/members` — Bot 获取群成员列表

但这些 API 返回的信息**不够丰富**，无法支撑龙虾协作场景的需求：
- 群列表没有返回群的类型（普通群 vs Space 群）、创建时间、成员数量等
- 成员列表没有区分人类和 Bot（`robot` 字段虽然有但未暴露给 Bot API）
- 没有 Bot 的描述信息（skill.md 内容、能力标签）
- 没有成员的在线状态和最后活跃时间

### 2.2 目标

增强 Bot 的群组查询 API，让 Bot 能够：
1. 查看自己所在的所有群的详细信息（含群类型、成员数、创建时间）
2. 查看群成员的详细信息（含是否 Bot、Bot 描述、在线状态）
3. 根据条件筛选群（如只看 Space 群、只看有其他 Bot 的群）

### 2.3 需要回答的设计问题

1. **API 设计**
   - 是扩展现有 `GET /v1/bot/groups` 的返回字段，还是新增 `GET /v1/bot/groups/detail` 端点？
   - 群列表是否支持分页？筛选条件怎么传？query parameter 还是 POST body？
   - 成员列表的扩展字段放在现有 `GET /v1/bot/groups/:group_no/members` 里，还是新端点？

2. **权限模型**
   - Bot 能看到群内其他 Bot 的 skill.md 吗？这算不算敏感信息？
   - Bot 能看到人类成员的在线状态吗？隐私边界在哪？
   - Bot 只能查自己所在的群，还是可以查任意群（如果知道 group_no）？

3. **数据来源**
   - 群类型信息从 `group` 表还是 `space` 表获取？
   - Bot 描述信息从 `robot` 表的哪个字段？`description`？还是从 skill.md API 拉？
   - 在线状态从 WuKongIM 的连接状态获取？还是从 heartbeat 时间推算？

4. **性能考量**
   - 如果一个 Bot 在 100 个群里，群列表 API 需要聚合多少数据？
   - 成员详情是否需要缓存？Bot 描述信息多久更新一次？
