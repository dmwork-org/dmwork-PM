## 一、实验目标

**核心命题**：彭特兰教授在人类团队中的发现——"全连接脉冲型组织在复杂协作任务上优于中心辐射型组织"——在AI Agent团队上是否成立？

**可验证假设**：在同构的复杂Feature开发任务中，采用全连接脉冲型（模式B）的AI Agent团队，比采用中心辐射型（模式A）的团队，产出更完整的方案、消耗更少的沟通成本。

## 二、实验设计

### 2.1 自变量（唯一改变的东西）

组织形式：模式A vs 模式B（详见第三节）

### 2.2 控制变量（保持不变的东西）

| 变量 | 控制方式 |
|------|---------|
| 龙虾团队 | 两轮使用完全相同的执行龙虾（Sensor/Designer + Builder + Reviewer） |

| 工具权限 | 相同的GitHub/代码库访问权限 |
| 时间预算 | 每轮12小时 |
| 品鉴者 | 同一位品鉴者（辉哥）做最终盲评 |
| 背景知识 | 两轮的龙虾都可访问Octo文档 |

### 2.3 因变量（我们测量的东西）

**主要指标（自动采集）**：

| 指标 | 定义 | 采集方式 |
|------|------|---------|
| 完成时间 | 从任务发布到产出完整方案的耗时 | 消息时间戳 |
| 消息总量 | 实验群内的总消息条数 | 消息API |
| 消息效率 | 有效产出 / 消息总量 | 人工+自动 |
| 话语权Gini系数 | 各龙虾发言量的均匀程度（0=完全均匀，1=一人独占） | 消息API统计 |
| @网络密度 | @mention关系的密度（实际@次数 / 可能@次数） | 消息解析 |
| 信息传递链长度 | 一个关键决策从提出到被所有相关龙虾确认，经过的消息数 | 人工标注 |
| PM瓶颈延迟（仅模式A） | 信息到达PM后到PM转发给目标龙虾的平均时间差 | 消息时间戳 |

**辅助指标（品鉴者评估）**：

| 指标 | 评估方式 |
|------|---------|
| 方案完整度 | 辉哥对两轮方案做盲评排序（不告知哪个是A哪个是B） |
| 方案可执行性 | 辉哥判断"哪个方案拿去直接开发，返工更少" |

## 三、两种组织形式

### 模式A：中心辐射型（Hub-and-Spoke）

**理论来源**：彭特兰教授在call center和银行团队研究中观察到的常见模式。所有信息通过一个中心节点（PM）中转。

**角色配置（4只龙虾）**：

| 角色 | 职责 | 沟通权限 |
|------|------|---------|
| **PM（项目经理）** | 接收所有信息、分配任务、协调进度、转发沟通 | 可以@任何人 |
| **Sensor/Designer** | 理解需求、定义场景、输出设计方案 | **只能@PM** |
| **Builder** | 评估技术可行性、输出技术方案和伪代码 | **只能@PM** |
| **Reviewer** | 审查方案完整性和一致性 | **只能@PM** |

**信息流规则**：

```
         Sensor/Designer
              ↕
Builder ←→   PM   ←→ Reviewer
```

1. **所有龙虾之间的沟通必须经过PM中转。** Sensor有问题不能直接问Builder，必须发给PM，PM转发。
2. **PM是唯一有全局视野的角色。** PM决定什么信息转发给谁、什么时候转发、是否需要补充说明。
3. **PM可以主动拉群讨论。** 但即使在讨论中，龙虾之间的回应也需要通过PM确认后再转达（模拟层级审批）。
4. **PM负责进度管控。** PM决定任务阶段切换（"需求阶段结束，进入设计阶段"）。

**违规判定**：如果Sensor直接@Builder讨论技术问题（绕过PM），该消息视为无效，PM有权要求重新通过正式渠道发送。

---

### 模式B：全连接脉冲型（Pulsed Full-Connect）

**理论来源**：彭特兰教授在高绩效创意团队中观察到的最优模式——所有人可自由沟通，但有节奏性的全员同步脉冲。

**角色配置（3只龙虾）**：

| 角色 | 职责 | 沟通权限 |
|------|------|---------|
| **Sensor/Designer** | 理解需求、定义场景、输出设计方案 | **可以@任何人** |
| **Builder** | 评估技术可行性、输出技术方案和伪代码 | **可以@任何人** |
| **Reviewer** | 审查方案完整性和一致性 | **可以@任何人** |

**无PM角色。** 模式A中的PM龙虾在本轮不参与执行（担任观察员，只记录不发言）。

**信息流规则**：

```
Sensor/Designer ←→ Builder
       ↕               ↕
       └──→ Reviewer ←──┘
     任何人可以@任何人
```

1. **任何龙虾可以随时@任何其他龙虾。** Builder发现需求不明确，直接@Sensor问。Reviewer提前看到设计有风险，直接@Designer说。
2. **任务分四个阶段，每个阶段之间有一次强制全员同步（脉冲）**：

**阶段划分与同步点**：

| 阶段 | 主导角色 | 产出 | 同步点要求 |
|------|---------|------|-----------|
| 阶段1：需求理解 | Sensor/Designer | 用户故事+场景定义 | **同步点1**：全员各发一条消息确认需求理解是否一致。有分歧→当场讨论解决→达成共识后进入下一阶段 |
| 阶段2：方案设计 | Sensor/Designer | API spec+数据模型+权限方案 | **同步点2**：全员确认方案。Builder提技术约束，Reviewer提风险→解决后进入下一阶段 |
| 阶段3：技术实现 | Builder | 实现方案+核心伪代码 | **同步点3**：全员确认技术方案可行性 |
| 阶段4：审查 | Reviewer | review意见+修订建议 | 最终产出 |

3. **阶段内部自由沟通。** 不限制谁先说话、谁跟谁讨论。
4. **同步点是强制的。** 必须所有龙虾都明确发消息表态（同意/有异议+具体内容）后才能进入下一阶段。

## 四、实验载体（两个同构Feature）

> 由辉哥指定，基于 dmwork-PM 已有 Issue。

### A组任务（模式A · 中心辐射型）：Issue #70 — Bot API 支持邀请其他 Bot 进群

**需求描述**：
当前 Bot API 有基础的群组查询能力（`GET /v1/bot/groups`、`GET /v1/bot/groups/:group_no/members`），但 **Bot 无法主动邀请其他 Bot 加入群聊**。在龙虾协作场景下，一个 Bot 需要拉其他 Bot 进群组建协作团队，目前只能靠人类手动操作。

**需要设计的内容**：
1. **API设计**：`POST /v1/bot/groups/:group_no/invite`（邀请端点）— 请求格式、批量支持、同步/异步返回
2. **权限模型**：谁有权邀请？Bot能邀请人类吗？被邀请Bot需要同意吗？不在线怎么办？
3. **与现有系统的交互**：
   - group模块：现有 `POST /v1/groups/:group_no/members` 是用户JWT鉴权，Bot Token鉴权怎么走？
   - WuKongIM：新成员加入后的频道订阅流程
   - events API：被邀请Bot怎么收到通知？
4. **群聊展示**：邀请成功后的系统消息（"Bot A 邀请 Bot B 加入群聊"），ContentType定义

**涉及模块**：botfather、group、WuKongIM、dmwork-adapters

**详细任务书** → [`ROUND-A-BRIEF.md`](ROUND-A-BRIEF.md)

---

### B组任务（模式B · 全连接脉冲型）：Issue #71 — Bot API 支持查询群列表和成员详情

**需求描述**：
当前 Bot 群组查询 API 返回的信息**不够丰富**——群列表没有群类型、创建时间、成员数量；成员列表没有区分人类和Bot（`robot`字段未暴露）、没有Bot描述信息、没有在线状态。龙虾协作需要Bot能充分了解自己所在的群和群内成员。

**需要设计的内容**：
1. **API设计**：扩展现有端点 vs 新增端点？分页方案？筛选参数（只看Space群、只看有Bot的群）？
2. **权限模型**：Bot能看其他Bot的skill.md吗？能看人类在线状态吗？只能查自己在的群还是任意群？
3. **数据来源**：
   - 群类型从group表还是space表？
   - Bot描述从robot表的description还是skill.md API？
   - 在线状态从WuKongIM连接状态还是heartbeat推算？
4. **性能考量**：Bot在100个群时的聚合查询性能、成员详情缓存策略

**涉及模块**：botfather、group、robot、space

**详细任务书** → [`ROUND-B-BRIEF.md`](ROUND-B-BRIEF.md)

---

### 同构性验证

| 维度 | #70 Bot邀请Bot进群 | #71 Bot查询群详情 |
|------|-------------------|------------------|
| 核心能力 | Bot群管理-写操作 | Bot群管理-读操作 |
| 新增/改造API端点 | 1-2个新端点 | 2-3个端点扩展 |
| 权限模型设计 | 邀请权限、Bot同意机制 | 信息可见性、隐私边界 |
| 跨模块依赖 | botfather+group+WuKongIM | botfather+group+robot+space |
| 关键设计决策点 | 拒绝处理、超时、批量 | 分页、筛选、缓存、隐私 |
| 需要多角色协作 | ✅ | ✅ |

## 六、预期结果与解读

| 结果 | 解读 |
|------|------|
| 模式B在多数指标上优于模式A | 彭特兰的人类实验结论在AI Agent上成立——全连接脉冲型更适合复杂协作任务 |
| 模式A在多数指标上优于模式B | AI Agent与人类不同——可能因为PM的信息过滤能力极强（不疲劳、不遗漏），中心辐射型反而更高效 |
| 两种模式无显著差异 | 组织形式可能不是AI Agent协作效率的主要瓶颈——需要寻找其他变量 |
| 模式A的Gini系数显著高于模式B | 中心辐射型导致话语权集中（PM主导），验证了彭特兰的参与度理论 |
| 模式B产出更快但质量更低 | 全连接模式可能导致"快速但粗糙"的共识——信息过多反而降低决策质量 |

## 七、已知局限

1. **样本量为1**：两轮实验各跑一次，无法做统计显著性检验。结论是探索性的，不是确定性的。
2. **顺序效应**：下半场的龙虾经历了上半场，可能"更熟练"。我们通过使用不同的实验群和不同的Feature来缓解，但不能完全消除。
3. **任务不完全相同**：虽然两个Feature结构同构，但具体的设计决策点不同。效率差异可能部分来自任务本身。
4. **模式A多一只龙虾（PM）**：这意味着模式A有更多总算力。如果模式A效率更高，部分原因可能是"多一个人"而非"组织形式更好"。

**缓解措施**：在分析报告中明确标注这些局限，不做过度因果推断。将本次实验定位为"探索性先导实验（pilot study）"，后续可基于本次发现设计更严格的对照实验。

## 八、后续行动

实验结束后，根据数据分析结果：

1. **如果模式B胜出**：将"阶段性全员同步脉冲"机制写入LOBSTER-ROLES协作规范，作为龙虾协作的标准流程。
2. **如果模式A胜出**：评估在现有龙虾团队中引入"PM龙虾"角色的可行性，更新角色分配表。
3. **无论哪种胜出**：将实验数据贡献给OCTO-ONA分析系统，作为AI Agent组织行为的基准数据集。

## 附录：数据采集脚本需求

需要一个cron job，每5分钟执行一次：

```
输入：实验群的channel_id（两个群）
调用：POST /v1/bot/messages/sync
输出：JSON文件，每条消息包含：
  - message_id
  - sender_id
  - sender_name（如果有）
  - timestamp
  - content（文本内容）
  - mention_uids[]（@了谁）
  - content_type
存储：按群和时间分文件，如 exp-a-2026-03-20-1400.json
```
