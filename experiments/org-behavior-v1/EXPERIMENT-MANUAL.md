## 一、实验目标

**验证命题**：彭特兰教授在人类团队上的发现——"全连接脉冲型组织在复杂任务上优于中心辐射型组织"——在AI Agent团队上是否成立？

**具体问题**：给同一组AI Agent两种不同的沟通规则，完成结构同构的复杂Feature，哪种组织形式产出更好、效率更高？

---

## 二、实验设计

### 2.1 自变量（唯一改变的东西）

**组织沟通模式**，两种：

| 维度 | 模式A：中心辐射型 | 模式B：全连接脉冲型 |
|------|------------------|-------------------|
| 信息路由 | 所有信息经PM中转 | 任何人直接找任何人 |
| 瓶颈 | PM是单点瓶颈 | 无瓶颈 |
| 信息过滤 | PM过滤和翻译信息 | 无过滤，原始信息直达 |
| 同步机制 | PM决定何时同步谁 | 阶段性全员强制同步 |
| 额外角色 | 需要PM角色 | 不需要 |

### 2.2 控制变量（保持不变的东西）

- 同一组执行龙虾（Sensor/Designer、Builder、Reviewer）
- 同构的任务（都是Bot与群聊交互能力扩展）
- 同样的代码库和工具访问权限
- 同样的品鉴者
- 同样的目标产出要求

### 2.3 任务选择

**A组任务（模式A · 中心辐射型）：Issue #70 — Bot API支持邀请其他Bot进群**

详细需求见 → [`ROUND-A-BRIEF.md`](ROUND-A-BRIEF.md)

涉及的设计决策：
- 权限模型：谁有权邀请Bot？Bot能拒绝吗？被邀请Bot不在线怎么办？
- API设计：`POST /v1/bot/groups/:group_no/invite` — 请求格式、批量支持、错误处理
- 群成员管理逻辑改动（group模块 — 现有邀请接口是用户JWT鉴权，Bot Token怎么走？）
- WuKongIM交互：新成员频道订阅流程
- 事件通知：被邀请Bot通过events API收到通知
- 群聊系统消息展示
- 跨模块：botfather + group + WuKongIM集成

**B组任务（模式B · 全连接脉冲型）：Issue #71 — Bot API支持查询群列表和成员详情**

详细需求见 → [`ROUND-B-BRIEF.md`](ROUND-B-BRIEF.md)

涉及的设计决策：
- API设计：扩展现有端点 vs 新增端点？分页？筛选参数？
- 权限模型：Bot能看到其他Bot的skill.md吗？能看人类成员在线状态吗？
- 数据来源：群类型从group表还是space表？Bot描述从robot表哪个字段？
- 性能考量：群列表聚合查询、成员详情缓存策略
- 跨模块：botfather + group + robot + space

**为什么同构**：都是"给Bot增加群管理能力"，都涉及权限设计+API设计+group模块改动+多个开放性设计决策，都需要多角色频繁沟通才能确定方案。

## 三、龙虾分配

### 执行龙虾（两轮不变）

| 角色 | 职责 |
|------|------|
| **Sensor/Designer** | 理解需求、定义用户场景、设计API和权限方案 |
| **Builder** | 编写技术方案、核心逻辑伪代码、确认可行性 |
| **Reviewer** | 审查方案一致性、安全性、与现有架构的兼容性 |

| **PM（项目经理）** | 接收所有信息、中转分发、协调进度、决定优先级 |

> PM龙虾在模式B中**不参与**，确保两轮的执行龙虾完全一致。

### 观察龙虾（两轮不变，不参与执行）

| 角色 | 职责 |

| **福柯** | 定性观察（话语权分布、权力动态、信息流模式） |

## 四、沟通规则详细定义

### 4.1 模式A：中心辐射型（上半场）

**核心规则：所有龙虾之间的沟通必须经过PM中转。禁止执行龙虾之间直接@对方。**

具体执行：

1. **任务启动**：PM在群里发布任务，@Sensor/Designer 开始需求理解
2. **需求→设计**：Sensor/Designer完成需求理解后，把产出发给PM（@PM）。PM审核后转发给Builder（@Builder），附上优先级判断和补充说明
3. **设计→实现**：Builder有问题不直接问Designer，而是@PM。PM去找Designer确认后回复Builder
4. **实现→审查**：Builder完成技术方案后发给PM。PM转发给Reviewer
5. **审查反馈**：Reviewer的意见发给PM，PM决定哪些打回重做、哪些通过，然后@相关龙虾

**信息流图**：
```
Sensor/Designer ←→ PM ←→ Builder
                    ↕
                 Reviewer
```

**PM的职责边界**：
- ✅ 转发信息、添加上下文、协调优先级、决定阶段切换
- ❌ 不自己做设计、不自己写代码、不自己做review

**违规处理**：如果执行龙虾之间直接@了对方（绕过PM），由观察龙虾记录为"违规事件"，纳入数据分析。

---

### 4.2 模式B：全连接脉冲型（下半场）

**核心规则：任何龙虾可以@任何龙虾自由沟通，但任务分四个阶段，每个阶段之间有一次强制全员同步。**

#### 阶段划分

**阶段1：需求理解**
- 主导者：Sensor/Designer
- 其他龙虾可以随时插嘴提问、挑战、补充
- 产出：用户故事 + 场景定义 + 边界条件

**→ 同步脉冲1**：全员各发一条消息，格式：
```
[同步1] 我对需求的理解：
- 核心场景：...
- 边界条件：...
- 未决问题：...
```
有分歧当场讨论解决，全员确认后进入阶段2。

**阶段2：方案设计**
- 主导者：Sensor/Designer
- Builder可以直接提技术约束（"这样设计后端改动太大"）
- Reviewer可以提前指出安全风险
- 产出：API spec + 数据模型 + 权限方案

**→ 同步脉冲2**：全员各发一条消息，格式：
```
[同步2] 我对方案的评估：
- 同意的部分：...
- 有疑虑的部分：...
- 建议修改：...
```
有分歧当场讨论解决，全员确认后进入阶段3。

**阶段3：技术实现方案**
- 主导者：Builder
- 可以随时@Designer确认业务细节
- 可以随时@Reviewer提前review关键决策点
- 产出：实现方案 + 核心逻辑伪代码 + 涉及的模块和文件列表

**→ 同步脉冲3**：全员各发一条消息，格式：
```
[同步3] 技术方案确认：
- 可行性判断：...
- 风险点：...
- 遗留问题：...
```

**阶段4：审查**
- 主导者：Reviewer
- 有问题直接@Builder和Designer讨论
- 产出：review意见 + 修订建议 + 最终确认

**信息流图**：
```
Sensor/Designer ←→ Builder ←→ Reviewer
      ↕                          ↕
      └──────────────────────────┘
           任何人可@任何人
         阶段间有同步脉冲
```

## 五、目标产出要求（两轮相同）

每轮必须产出以下内容，作为评估对象：

1. **用户故事文档**：谁、在什么场景、要什么、为了什么
2. **API设计方案**：端点、请求/响应格式、错误处理
3. **权限模型**：谁能调用、怎么鉴权、边界条件
4. **技术实现方案**：涉及哪些模块、核心逻辑伪代码、数据库变更（如有）
5. **Review意见**：方案的问题、风险、改进建议

## 七、数据采集与评估指标

### 7.1 自动采集指标（定量）

| 指标 | 定义 | 采集方式 |
|------|------|----------|
| **完成时间** | 从任务发布到最终产出物提交的时长 | 消息时间戳 |
| **总消息数** | 群内所有消息数量 | 消息API |
| **消息效率** | 总消息数 / 产出文档section数 | 手动统计 |
| **Gini系数** | 话语权均匀度（0=完全均匀，1=一人独占） | 按发送者统计消息量计算 |
| **@mention网络密度** | 实际@连接数 / 可能的最大@连接数 | 解析@mention |
| **响应延迟** | 被@后到回复的平均时间 | 消息时间戳差 |
| **信息传递链长度** | 一个关键决策从提出到被所有相关方知道，经过几条消息 | 人工标注3-5个关键决策点 |
| **阶段切换时间** | 从一个阶段结束到下一阶段开始的间隔 | 消息时间戳 |

### 7.2 定性观察指标（福柯负责）

- 谁主导了话语权？信息是如何流动的？
- 模式A中PM的信息过滤行为：过滤了什么？遗漏了什么？
- 模式B中是否自发涌现了某种协调角色？
- 两种模式下Reviewer介入的时机和深度有何不同？
- 龙虾是否违反了沟通规则？违反的动机和效果是什么？

### 7.3 品鉴者评估（娃爹，30秒即可）

两轮产出物匿名呈现（标为"方案X"和"方案Y"），回答：

1. 哪个方案更完整？
2. 哪个方案更可执行？
3. 如果只能选一个立刻实施，选哪个？

## 八、判断标准

| 结果 | 含义 |
|------|------|
| 模式B在完成时间、消息效率、方案质量上均优于模式A | 彭特兰的结论在AI Agent上成立：全连接脉冲型优于中心辐射型 |
| 模式A反而更优 | AI Agent场景下PM中转有独特价值（可能因为AI处理速度快，消除了人类PM的瓶颈效应） |
| 两者无显著差异 | 对AI Agent来说沟通拓扑可能不是关键变量，需要探索其他维度 |
| 模式B方案质量高但耗时更长 | 全连接带来了更充分的讨论但也带来了更多沟通开销，需要调整脉冲频率 |

## 九、风险与应对

| 风险 | 应对 |
|------|------|
| 模式A中龙虾绕过PM直接@对方 | 由观察龙虾记录为违规事件，纳入分析（本身也是有价值的数据） |
| 两个Issue复杂度不完全对齐 | 事后通过产出物的section数和决策点数量校准 |
| 龙虾清空上下文后对DMWork系统理解丧失 | 下半场开始时给10分钟让龙虾重新读关键文档（Octo/Bot系统/group模块） |
| 品鉴者评估的主观性 | 匿名呈现 + 只问简单的比较问题，不做打分 |
