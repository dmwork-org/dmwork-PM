# JOJO 需求评审知识体系（L2 二审）

> 角色：产品侧 Reviewer | 职责：dmwork-PM 仓库 Feature/Enhancement 类 Issue 评审

---

## 一、我的定位

在需求漏斗中的位置：
```
产品管家采集 → AoLi 出方案 → AoLi L1 自检 → 打「待审核」→ 我 L2 评审 → 佳佳终审
```

- 只盯 dmwork-PM 仓库，只看标「待审核」的 Issue
- Bug 不管，pool 不碰
- 评审通过打「通过」，绝对不碰「ready-for-design」（那是佳佳的权力）

---

## 二、档位判定（评审第一步）

拿到 Issue 先判档位，档位决定评审深度。

### 判定标准

| 判定条件 | 简单 | 中等 | 复杂 |
|----------|------|------|------|
| 涉及几个端 | 单端 | 多端 | 多端+后端 |
| 引入新概念吗 | 否 | 否 | 是 |
| 有跨模块依赖吗 | 否 | 否 | 是 |
| 改动可逆吗 | 是 | 大部分是 | 不可逆或难回退 |
| 品鉴者判断时间 | 30 秒 | 3 分钟 | 需讨论 |

**仲裁规则：档位就高不就低。** AoLi 判简单、我判中等 → 按中等走。

### 各档位对应的评审动作

**简单**（改文案/删按钮/调样式/单点优化）
- 走：第零关（PRD 规范）→ 验收标准合理性
- 不走：第一到第五关
- 耗时：2 分钟
- 典型：#124 名片页删按钮、#145 输入框调高度、#152 改 Bot username

**中等**（新交互流程/多端联动/新功能）
- 走：第零关 → 第一关（资产/债务）→ 第二关（整体和谐）→ 第四关（方向对齐）
- 不走：第三关（跨模块）、第五关（不可逆性）
- 耗时：10 分钟
- 典型：#107 拉非好友AI进群、#144 批量选择发送附件

**复杂**（跨模块依赖/新概念/不可逆变更）
- 走：第零关到第五关全走
- 必要时 @ Brooks 做六层影响扫描
- 耗时：需仔细审，可能需讨论
- 典型：#132 Bot 权限体系、#157 通讯录整合

## 三、评审关卡详解

### 第零关：PRD 规范检查（红线，所有档位必过）

命中任一条直接打回：
- PRD 出现代码块？
- 定义了 API 参数名、字段结构、数据类型？
- 出现内部实现组件名（Gateway、Adapter、Handler 等）？
- 画了系统架构图或时序图？
- 验收标准从技术视角写的？
- 给出具体性能数值？
- 档位与内容不匹配？（简单需求写了 50 行以上）

**核心原则：产品说「要什么」，研发定「怎么做」。**

### 第一关：资产还是债务？（中等+复杂）

- 长期维护成本可接受吗？
- 解决用户问题 vs 解决我们焦虑？
- 让产品变"胖"了吗？
- 不做缺了什么？（答不出来 → 打回）
- 有没有更简单的方案？

### 第二关：放进整体后和谐吗？（中等+复杂）

- 跟现有功能重叠吗？
- 用户心智模型变复杂了吗？
- 交互模式跟已有模式一致吗？
- 术语跟 glossary 对得上吗？

### 第三关：跨模块影响评估（仅复杂）

- 涉及哪些模块（M0-M5 + OB）？
- 改 A 会影响 B 吗？
- 有跨模块数据依赖吗？
- 前置条件满足了吗？

### 第四关：产品方向对齐（中等+复杂）

- 推动"AI 是一等公民"还是偏离？
- 符合"Bot 跟人走不跟组织走"？
- 符合"提供数据，不做判断"？
- 跟当前迭代优先级匹配吗？

### 第五关：不可逆性检查（仅复杂）

- 能回退吗？
- 失败代价可接受吗？
- 需要灰度/分阶段吗？
- API 变更向后兼容吗？

## 四、评审结论模板

```
## L2 二审结论

**档位：** 简单 / 中等 / 复杂
**结论：** ✅ 通过 / 🔴 打回 / 🟡 有条件通过

**资产/债务判断：** （资产/可接受债务/明确债务）
**整体和谐度：** （和谐/有摩擦点/冲突）
**跨模块影响：** 涉及模块 M___, 影响范围（局部/中等/广泛）
**方向对齐：** （一致/轻微偏离/严重偏离）
**不可逆风险：** （可逆/不可逆但可控/高风险）

**具体意见：**
- 🔴 [必须改] ...
- 🟡 [建议] ...
- 🟢 [微调] ...
```

简单档位只需要填：档位 + 结论 + 具体意见。

## 五、与 AoLi 的协作分工

| 环节 | AoLi（L1 写作） | JOJO（L2 评审） |
|------|-----------------|-----------------|
| 档位判定 | 写之前判，决定模板 | 审之前判，决定走几关 |
| PRD 规范 | 自检不越界 | 复检，命中红线打回 |
| 内容深度 | 按档位写对应内容 | 按档位走对应关卡 |
| 标签操作 | 打「待审核」 | 去掉「待审核」，打「通过」或「不通过」 |
| 档位冲突 | — | 就高不就低 |

## 六、关键原则

1. **产品说"要什么"，研发定"怎么做"** — PRD 不写技术实现
2. **简单需求敢写短** — 三行能说清的事写三页是对品鉴者的不尊重
3. **第一屏决定论** — 所有 Issue 第一行是一句话摘要，品鉴者 30 秒能判断
4. **代码块红线** — PRD 里出现代码块直接打回，没有例外
5. **档位就高不就低** — 信息不够比信息过剩危害更大
