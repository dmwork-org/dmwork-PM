# L2 二审 Checklist（JOJO）

> 核心问题：这个功能**放进 Octo 整体后，系统还和谐吗？是资产还是债务？**
> 一审已确认痛点真实、方案合理。二审关注的是全局影响。

## 前置：档位判定

评审前先判断需求档位，档位决定走哪些关卡。

**判定标准：**

| 判定条件 | 简单 | 中等 | 复杂 |
|----------|------|------|------|
| 涉及几个端 | 单端 | 多端 | 多端+后端 |
| 引入新概念吗 | 否 | 否 | 是 |
| 有跨模块依赖吗 | 否 | 否 | 是 |
| 改动可逆吗 | 是 | 大部分是 | 不可逆或难回退 |
| 品鉴者判断时间 | 30 秒 | 3 分钟 | 需讨论 |

**仲裁规则：档位就高不就低。** AoLi 判简单、我判中等 → 按中等走。

**各档位走哪些关卡：**
- **简单**：第零关（PRD 规范）→ 验收标准是否合理 → 出结论。2 分钟内。
- **中等**：第零关 → 第一关（资产/债务）→ 第二关（整体和谐）→ 第四关（方向对齐）→ 出结论。10 分钟内。
- **复杂**：第零关到第五关全走（第三关为路由决策点，见下）。

## 串行漏斗（依次过筛）

### 第零关：PRD 规范检查（红线，所有档位必过）

> 命中任一条直接打回，不进后续评审。参考 [PRD-STANDARD.md](https://github.com/dmwork-org/dmwork-PM/blob/main/PRD-STANDARD.md)

- [ ] PRD 中是否出现代码块（```json、```yaml 等）？
- [ ] PRD 中是否定义了 API 参数名、字段结构、数据类型？
- [ ] PRD 中是否出现内部实现组件名（Gateway、Adapter、Handler 等）？
- [ ] PRD 中是否画了系统架构图或时序图（而非用户操作流程图）？
- [ ] 验收标准是否从技术视角而非用户视角写的？
- [ ] 是否给出了具体性能数值（而非产品语言的体验描述）？
- [ ] 档位与内容是否匹配？（简单需求写了 50 行以上 → 打回要求精简）

> ❌ 命中任一条 → 打回，注明"PRD 越界到技术方案，请参考 PRD-STANDARD.md 修改"

### 第一关：资产还是债务？（Ken Norton）（中等+复杂）

- [ ] 加完这个功能后，长期维护成本可接受吗？
- [ ] 是在解决用户的问题，还是在解决我们自己的焦虑？
- [ ] 这个功能会不会让产品变"胖"——用户要学习的概念又多了一个？
- [ ] 不做这个功能，产品缺了什么？（如果答不出来 → 🔴）
- [ ] 有没有更简单的方案达到同样效果？能不能删掉一半？

> ❌ 明确是债务（维护成本高、用户价值低）→ 直接打回

### 第二关：放进整体后和谐吗？（Christopher Alexander）（中等+复杂）

- [ ] 跟现有功能/模块有没有重叠？有没有已经在做类似事情的功能？
- [ ] 用户的心智模型会因此变复杂吗？（用户需要理解的概念数量是否增加）
- [ ] 这个功能的交互模式跟产品已有模式一致吗？还是引入了新范式？
- [ ] 命名/术语跟 glossary 对得上吗？有没有引入新的概念但没定义？

> ❌ 跟现有功能矛盾或重叠 → 打回，要求合并或重新定位

### 第三关：跨模块路由决策（仅复杂）

> ⚡ 这一关是**路由决策点**，不是 JOJO 自己做跨模块分析。
> 快速判断是否跨模块（10 秒内），然后决定路由。

**快判：这个需求是否涉及 2 个及以上模块（M0-M5 + OB）？**

- **不跨模块** → JOJO 自己过，直接进第四关
- **跨模块** → 在群 c2b8178022e54523ae9ad32d62ca4f52 里 @ Brooks，发 Issue 链接，请求六层影响扫描。等 Brooks 返回结果后，将扫描结论纳入评审意见（模块归属、跨层依赖、接口契约建议）

**为什么不自己做跨模块分析：** Brooks 有六层影响扫描能力（实体/信息/管道/过程/治理/可观测），比 JOJO 粗判更精准，避免两人重复劳动。

> ❌ 跨模块但未经 Brooks 扫描 → 不出结论，等扫描结果

### 第四关：产品方向对齐（中等+复杂）

- [ ] 跟 Octo 北极星指标一致吗？推动"AI 是一等公民"还是偏离了？
- [ ] 符合"Bot 跟人走不跟组织走"的核心原则吗？
- [ ] 符合"提供数据，不做判断"的平台设计原则吗？
- [ ] 跟当前迭代的优先级匹配吗？是在做当下该做的事吗？

> ❌ 方向偏离产品原则 → 打回，标注具体偏离了哪条原则

### 第五关：不可逆性检查（仅复杂）

- [ ] 这个决策做了以后能回退吗？
- [ ] 如果不能回退，失败的代价可接受吗？
- [ ] 需不需要灰度/分阶段上？
- [ ] 数据结构/API 变更是否向后兼容？

> ❌ 不可逆且风险高 → 打回或要求分阶段实施

## 二审输出原则

**模板内化，输出差异化。** 关卡是思考框架，不是输出格式。

评审结论只写两样东西：**结论 + 这个需求我具体看到了什么。**
- 不逐关卡填表，不写通用套话
- 每个需求的通过/打回理由必须是具体的、不同的
- 写对品鉴者做判断有用的信息：风险在哪、建议是什么、跟哪个 Issue 有联动

**好的评审：**
> #152：✅ 通过。username 改了会影响全平台 @ 标识，建议加冷却期。注意跟 #71 群列表缓存一致性。

> #162：✅ 通过。自动记忆回写解决了 prompt 指导失效的问题，关注回写内容的信噪比——垃圾进垃圾出。

> #145：✅ 通过。简单 UI 改进，没什么风险。

**坏的评审（不要这样写）：**
> ✅ 通过。资产。用户价值清晰，维护成本可控。和谐。方向一致。

这种结论对 15 个 Issue 都适用 = 对哪个都没用。
