# Feature Spec：消息理解层（Message Understanding Layer）

> **优先级**：P0
> **覆盖痛点**：N013 语音消息处理缺失、N017 不可解析消息标准化回退、N018 消息时序感知
> **一句话**：在消息到达 Agent 之前，把富媒体转换成 Agent 可理解的结构化文本，消除信息有损转换。

---

## 1. 问题定义

### 1.1 现状

人类发送的消息是富媒体的（语音、图片、文件、引用、表情），但 Agent 收到的是降级后的纯文本占位符：

| 消息类型 | 人类发送 | Agent 实际收到 | 信息损失 |
|---------|---------|---------------|---------|
| 语音 | 45秒战略分析 | `[语音消息] https://cos.xxx/audio.wav` | 100%（Agent 拿到 URL 但没法听） |
| 图片 | 产品架构截图 | `[图片] https://cos.xxx/img.png` | ~90%（URL 可下载但大部分框架不处理） |
| 图片+配文 | 截图+"你看这个bug" | 只收到图片 URL，配文丢失 | 配文 100% 丢失 |
| 引用回复 | 回复张三的消息 | 引用关系丢失 | Agent 不知道在回复谁 |
| 文件 | PDF 报告 | `[文件] report.pdf` | 内容 100% 不可读 |
| 表情/贴纸 | 自定义表情 | 可能丢失或变成乱码 | 语义丢失 |

### 1.2 根因

Adapter 层的 `resolveContent()` 函数只做了最简单的类型判断和文本拼接，没有对非文本消息做语义转换。这不是 bug，是功能缺失。

### 1.3 影响

- 辉哥是语音重度用户，所有 Bot 只收到占位符 → 沟通效率直接砍半
- 图片配文丢失（Web 端图片+配文发送后 Bot 只收到图片） → 上下文不完整
- 引用关系丢失 → 多轮对话中 Agent 不知道在讨论哪条消息

---

## 2. 产品方案：消息预处理管道

### 2.1 架构

在 Adapter 的 `resolveContent()` 之后、消息投递给 Agent 之前，插入一个**消息预处理管道（Message Preprocessing Pipeline）**：

```
消息进入 Adapter
    ↓
[1. 类型识别] — 判断消息类型（文本/语音/图片/文件/引用/表情/混合）
    ↓
[2. 转换管道] — 按类型分别处理
    ├── 语音 → STT 转写
    ├── 图片 → Vision 描述 / OCR
    ├── 文件 → 内容提取
    ├── 引用 → 查找原始消息
    └── 混合 → 各部分分别处理后合并
    ↓
[3. 结构化输出] — 生成 Agent 可理解的富文本
    ↓
投递给 Agent
```

**核心原则**：
- **每个环节独立** — 语音 STT 失败不影响图片处理
- **失败不阻断** — 任何转换失败都 graceful degradation 回退到原始格式
- **可配置** — 每个能力可按 Space/Bot 粒度开关
- **异步不阻塞** — 转换耗时长的（如长语音 STT）异步处理，先投递可解析部分

### 2.2 语音处理

#### 输入
```
消息类型: voice
音频 URL: https://cos.xxx/audio.wav
时长: 45秒
```

#### 处理流程
```
下载音频 → 格式检测 → STT 转写 → 质量校验 → 结构化输出
```

#### 输出（Agent 收到的格式）
```
[语音转写] 我觉得这个方案有三个问题：第一是成本太高，我们的预算只有50万...
（时长: 45秒 | 语言: zh-CN | 置信度: 0.92）
```

#### 设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| STT 引擎 | Whisper API（OpenAI）或自部署 Whisper | 多语言支持好，准确率高 |
| 长语音（>3分钟） | 分段转写 + 整体输出 | 避免单次 API 超时；不做自动摘要（摘要是语义层的事，不该在预处理管道做） |
| 转写失败 | 回退到 `[语音消息，转写失败] 原始音频: URL` | 不阻断，Agent 至少知道有条语音消息 |
| 多语言 | Whisper 自动检测，输出标注语言 | 不需要预设语言 |
| 成本控制 | 按 Space 配置开关，默认开启 | 私有部署可选自部署 Whisper 降低成本 |
| 转写结果存储 | 缓存到消息元数据，避免重复转写 | 同一条语音被多个 Bot 处理时只转写一次 |

### 2.3 图片处理

#### 输入
```
消息类型: image
图片 URL: https://cos.xxx/screenshot.png
配文: "你看这个bug"（可选，可能丢失）
```

#### 处理流程
```
下载图片 → 类型判断（截图/照片/架构图）→ Vision 描述 / OCR → 合并配文 → 结构化输出
```

#### 输出
```
[图片] "你看这个bug"
[图片描述] 一张网页截图，显示一个弹窗报错：TypeError: Cannot read property 'name' of undefined。页面标题为"Bot 设置"。
（原图: URL）
```

#### 设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 是否所有图片都做 Vision | 默认开启，可配置关闭 | Vision 成本低，价值大 |
| 配文合并 | 配文 + 图片描述合并输出 | 解决 Web 端配文丢失问题 |
| 图片过大 | 先压缩/缩放再处理 | 控制 API 调用成本 |
| Vision 引擎 | 复用 Adapter 已有的 LLM（带 Vision 能力） | 不需要额外接入 |
| 多图消息 | 每张图分别描述，按序号标注 | `[图片1] ... [图片2] ...` |

### 2.4 文件处理

#### 处理策略（按文件类型）

| 文件类型 | 处理方式 | 输出 |
|---------|---------|------|
| PDF | 提取前 N 字符文本 | `[文件: report.pdf] 内容预览：...（共 15 页，23,000 字）` |
| Word/Markdown | 提取全文或前 N 字符 | 同上 |
| 代码文件 | 提取全文 | `[代码文件: main.go] 内容：...` |
| 表格(CSV/Excel) | 提取前 N 行 | `[表格: data.xlsx] 列: A,B,C | 前5行: ...` |
| 其他/不可解析 | 元信息 | `[文件: video.mp4, 大小: 45MB, 类型: video/mp4] 该文件类型暂不支持内容提取` |

#### 设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 提取字符上限 | 默认 4000 字符，可配置 | 太多会撑爆上下文 |
| 大文件 | 只提取元信息 + 前 N 字符 | 控制 token 消耗 |
| 二进制文件 | 只输出元信息 | Agent 无法处理二进制 |

### 2.5 引用/回复处理

#### 现状
引用关系在 Adapter 层完全丢失，Agent 不知道消息在回复谁。

#### 处理流程
```
消息带 reply_to_message_id → 查找被引用的原始消息 → 拼接到当前消息前面
```

#### 输出
```
> [引用 张三 3分钟前]: "我觉得方案A更好，因为成本低30%"

李四: 我不同意，原因是市场窗口只有3个月，便宜但慢的方案来不及。
```

#### 设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 引用嵌套 | 最多展开 2 层（引用的引用） | 避免无限嵌套 |
| 被引用消息被删除 | `[引用 已删除的消息]` | 不阻断 |
| 引用长消息 | 截断到 200 字符 + "..." | 不撑爆上下文 |

### 2.6 表情/贴纸

| 类型 | 处理 |
|------|------|
| 标准 Unicode emoji | 直接传递 |
| 自定义 emoji | `[表情: emoji_name]` |
| 贴纸 | `[贴纸: sticker_name]`，如有图片描述则附加 |

### 2.7 混合消息

一条消息包含多种类型（如图片+配文+引用）时，按顺序拼接各部分的处理结果。

## 3. 配置体系

### 3.1 Space 级别配置

Space 管理员可以配置消息预处理管道的开关：

```yaml
space_settings:
  message_preprocessing:
    voice_stt: true          # 语音转写
    image_vision: true       # 图片描述
    file_extraction: true    # 文件内容提取
    reply_expansion: true    # 引用展开
    max_extract_chars: 4000  # 文件提取字符上限
```

### 3.2 Bot 级别覆盖

单个 Bot 可以覆盖 Space 配置（例如某个 Bot 不需要图片描述能力）：

```yaml
bot_settings:
  message_preprocessing:
    image_vision: false      # 这个 Bot 不需要看图
```

### 3.3 优先级
Bot 配置 > Space 配置 > 平台默认值

## 4. 边界条件

| 场景 | 处理方式 |
|------|---------|
| STT 服务不可用 | 回退到原始格式，不阻断消息投递 |
| Vision API 超时 | 先投递文本部分，图片描述异步追加（或放弃） |
| 文件过大（>50MB） | 只提取元信息，不下载内容 |
| 加密消息 | 跳过预处理，标注 `[加密消息，无法处理]` |
| 同一条语音被 5 个 Bot 处理 | 只转写一次，缓存复用 |
| 语音内容涉及敏感信息 | 转写后跟普通文本一样，走 Bot 的权限规则 |

---

## 5. 不做什么

- **不做语义摘要** — 管道只做格式转换，不做语义理解。"45秒语音的3个要点"是 Agent 自己的事
- **不做内容审核** — 管道不审查内容，审核是独立功能
- **不做翻译** — 管道不做语言翻译，Agent 自己处理
- **不改消息存储** — 原始消息不变，转换结果只用于 Agent 投递

---

## 6. 衡量指标

| 指标 | 目标 |
|------|------|
| 语音转写准确率 | >90%（中文场景） |
| 预处理延迟（P95） | 语音 <5秒，图片 <3秒，文件 <2秒 |
| 管道可用性 | >99.5%（单环节故障不影响整体） |
| 信息完整度提升 | 从当前 ~40% 提升到 >90% |
