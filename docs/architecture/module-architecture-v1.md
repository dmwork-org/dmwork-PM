# OCTO 模块架构定义 v1

> 三方收敛定稿：毕达哥拉斯 × wuyun3 × 彭特兰
> 日期：2026-03-19
> 用途：所有龙虾设计需求时的参照标准
> 使用方式：设计需求 → 确认影响哪个模块 → 读该模块定义 → 确认接口和边界 → 标注跨模块依赖 → 提需求

## 总览

```
外部接入层（不编号）
  DMWork客户端 / Web端 / Octic硬件 / 第三方接入
          │
          ▼
┌─────────────────────────────────────────────────┐
│  M0 · 身份与权限（Identity & Access）             │
├─────────────────────────────────────────────────┤
│                                                   │
│  M1 · 消息通道（Message Channel）                  │
│    含平台语义增强（@解析、引用还原、身份解析）       │
│                                                   │
│  M2 · Agent运行时（Agent Runtime）                 │
│    龙虾生命周期 + context window组装规则            │
│                                                   │
│  M3 · 知识存储（Knowledge Store）                  │
│    M3a 龙虾知识（OpenClaw侧）                      │
│    M3b 平台知识（Octo侧）                          │
│                                                   │
│  M4 · 品鉴信号（Connoisseur Signal）               │
│    采集 → 传递 → 记录 → 输出                       │
│                                                   │
│  M5 · 协作协议（Collaboration Protocol）            │
│    拓扑层 + 协议层 + 规范层                         │
│                                                   │
├─────────────────────────────────────────────────┤
│  OB · 可观测性（Observability）— 后置横切层         │
└─────────────────────────────────────────────────┘
```

### 设计原则

1. **提供数据，不做判断** — 平台层模块不替龙虾做业务决策
2. **按职责拆，不按代码仓库拆** — 同一模块可能跨Octo和OpenClaw两个代码库
3. **每个模块必须回答三个问题** — 接口（提供什么）、边界（不管什么）、依赖规则（别人怎么知道会影响你）
4. **知识外化优于参数化** — 所有知识在可读文本文件中，不在模型参数里（论文核心命题的工程保障）
5. **规范归属判断标准** — 只涉及一只龙虾自己→归M2；涉及两只以上龙虾交互→归M5

## M0 · 身份与权限（Identity & Access）

### 定义
管理所有实体（用户、龙虾、品鉴者）的身份识别和权限分级。回答"你是谁、什么级别、能做什么"。

### 接口
- 用户身份查询（UID → 名字、类型、品鉴者级别）
- 龙虾身份注册与查询（Bot Token、IDENTITY.md映射）
- 品鉴者分级管理（最高/架构级/模块级/普通用户）
- 权限校验（某操作是否需要某级别品鉴者批准）

### 边界（不管什么）
- 不管消息怎么传递（M1的事）
- 不管龙虾怎么运行（M2的事）
- 不管品鉴信号的内容（M4的事）
- 只管"谁有什么资格"，不管"资格怎么使用"

### 前置依赖
```
M0 能工作的前提：
└── Octo user表 + robot表（已存在）
```
M0是其他模块的前置条件，自身无前置依赖。

### 数据存储
- **Octo侧**：用户基础身份、品鉴者分级、Bot Token
- **OpenClaw侧**：龙虾的IDENTITY.md（龙虾自己的身份）

### 变更审批
辉哥（身份和权限是安全底线）

### 当前状态
**部分存在**。用户/龙虾基础身份已有（Octo user/robot表），品鉴者分级不存在（Octo源码中不区分品鉴者和普通用户）。v1目标：至少实现二级（品鉴者/普通用户）。

## M1 · 消息通道（Message Channel）

### 定义
所有消息和事件的收发路由基础设施。包含平台语义增强能力（@解析、引用还原、身份解析）。回答"消息怎么从A到B、这条消息的基本结构是什么"。

### 接口
- 发送消息（Bot API: sendMessage, deliver）
- 接收消息（webhook推送）
- 查询历史（sync API）
- 流式输出（stream start/end）
- **四类事件推送**（全部为一等公民）：
  1. 消息事件（文本/图片/语音/文件/卡片）
  2. 反应事件（emoji reaction 添加/移除）
  3. 成员变动事件（加入/退出群聊）
  4. 在线状态事件
- **平台语义增强**（原M1.5职责，现为M1子能力）：
  - @mention解析（UID → 显示名）
  - 引用关系还原（reply消息关联）
  - 消息sender身份注入（from_name, from_type）
  - @门控（群聊中未被@的消息缓存不触发）
  - 历史上下文注入（被@时注入最近N条缓存消息）
  - 出站处理（@名字反查为UID）

### 边界（不管什么）
- 不做协作语义解析（task_context结构化归M5）
- 不做品鉴信号识别（"这个reaction是品鉴还是随手点的"归M4）
- 不做消息内容的业务判断（"这是需求还是闲聊"归龙虾自己）
- 不管龙虾怎么处理收到的消息（M2的事）

### 前置依赖
```
M1 能工作的前提：
└── WuKongIM通信引擎（已运行）
```

### 归属
Octo侧（WuKongIM + BotFather API + DMWork Plugin平台语义部分）

### 变更审批
研发侧（Octo团队 + 水下雷达）

### 当前状态
**基本运行，缺关键能力**：
- ✅ 消息收发、@门控、历史注入
- ❌ reaction事件推送（M4的前置条件）
- ❌ 成员变动/在线状态事件推送
- ❌ from_name/from_type字段（靶子①，本周修）
- ❌ file_size字段（靶子④，本周修）
- ⚠️ stream API报400错误

## M2 · Agent运行时（Agent Runtime）

### 定义
龙虾的完整生命周期管理——从注册到运行到学习。回答"龙虾怎么活"。

### 接口
- Bot注册与管理（BotFather：注册、Token、命令菜单）
- Session管理（创建、切换、重置、compaction）
- 消息dispatch（收到消息 → 查找session → 触发推理）
- **Context window组装规则**（显式接口）：
  - 永久层文件加载策略（SOUL/IDENTITY/USER/TOOLS/AGENTS/HEARTBEAT必加载）
  - MEMORY.md条件加载（仅主session加载，群聊不加载 — 安全策略）
  - 对话历史截断策略（token预算分配）
  - 群聊历史注入量（默认20条，可按群活跃度调整 — 未来）
- 工具调用管理（哪些工具可用、权限控制）
- 子agent调度（sessions_spawn/sessions_send）
- Compaction（对话历史压缩）
- 大文件保护（基于M1提供的file_size做门控）

### 边界（不管什么）
- 不管消息怎么路由到自己（M1的事）
- 不管品鉴信号怎么采集（M4的事）
- 不管龙虾之间怎么协作（M5的事）
- 不管龙虾的知识怎么存储（M3的事，M2只负责"加载"）
- **只管单个龙虾自身的行为规范**，不管龙虾间的协作规范（M5的事）

### 前置依赖
```
M2 能工作的前提：
├── M0 提供Bot身份（BotFather注册）
├── M1 提供消息输入
└── M3a 提供知识文件
```

### 归属
跨两侧（BotFather在Octo侧，Gateway/Session在OpenClaw侧）

### 变更审批
黄楠 + OpenClaw维护者

### 当前状态
**运行中，有已知问题**：
- ✅ Session管理、context window组装、工具调用、子agent
- ❌ Session自动管理（满了需手动reset）
- ❌ 大文件保护（无file_size门控）
- ❌ context window组装规则未文档化

## M3a · 龙虾知识（Agent Knowledge）

### 定义
龙虾侧的所有知识存储和检索。存在OpenClaw文件系统中，龙虾自己读写。回答"龙虾记住了什么、怎么找到"。

### 接口
- 读写永久层文件（SOUL.md, MEMORY.md, memory/*.md, IDENTITY.md, USER.md, TOOLS.md, AGENTS.md）
- 读写Skill文件（skills/*/SKILL.md）
- 读写共享知识库（shared/目录）
- 语义搜索（memory_search）
- 精确读取（memory_get）

### 边界（不管什么）
- 不管知识怎么被加载到context window（M2的事）
- 不管平台侧的知识（Context Card、用户Profile归M3b）
- 不管品鉴信号（M4的事）
- 只管"存"和"找"，不管"怎么用"

### 前置依赖
```
M3a 能工作的前提：
└── OpenClaw文件系统（已存在）
```

### 关于shared/目录
**当前状态**：shared/是跨龙虾共享知识的临时方案，依赖"龙虾恰好在同一台机器上"的物理约束。
**终态**：跨龙虾共享知识应由M3b提供平台级支持，或M3a定义"共享知识同步协议"。
**当前归属**：M3a（物理上在OpenClaw文件系统）。标注为**M3b的临时降级实现**。

### 归属
OpenClaw侧

### 变更审批
各龙虾自治（个体知识）/ 团队共识（shared/）

### 当前状态
**运行中**：
- ✅ 个体知识读写、语义搜索
- ✅ shared/目录已建，10+文件
- ❌ 共享知识同步协议（跨机器场景）
- ❌ 知识版本历史（何时改、为什么改）

## M3b · 平台知识（Platform Knowledge）

### 定义
Octo平台侧的结构化知识存储。存在Octo数据库中，通过API读写。回答"平台知道关于龙虾和用户的什么"。

### 接口
- Context Card读写（GET/PUT /v1/bot/me/context）— 龙虾的公开身份名片
- 用户Profile读写 — 品鉴者偏好、Skill个性化配置、行为日志
- 品鉴行为日志存储（M4的输出存这里）
- 共享知识服务（未来：跨龙虾的平台级知识共享）

### 边界（不管什么）
- 不管龙虾私有知识（SOUL/MEMORY/Skill归M3a）
- 不管知识怎么被使用（M2的事）
- Context Card是**公开身份**，不是私有记忆

### 前置依赖
```
M3b 能工作的前提：
├── M0 提供用户/龙虾身份
└── Octo数据库（已存在）
```

### 归属
Octo侧

### 变更审批
Octo团队

### 当前状态
**大部分不存在**：
- ❌ Context Card（robot表无context_card字段，API未实现）
- ❌ 用户Profile（无独立Profile表）
- ❌ 品鉴行为日志存储
- ❌ 共享知识服务

## M4 · 品鉴信号（Connoisseur Signal）

### 定义
品鉴者反馈行为的完整生命周期管理。回答"品鉴者说了什么、怎么传给龙虾、怎么记录、输出什么格式"。

### 接口（四个动词）
1. **采集**：从用户自然行为中识别品鉴信号
   - 信号类型：emoji reaction、文字评价、直接修改产出物、直接改Memory
   - 信号强度谱系：emoji（最轻）→ 自然语言 → 改产出物 → 改Memory（最强）
2. **传递**：Octo侧把原始事件推给龙虾
   - 依赖M1的reaction/成员/状态事件推送能力
3. **记录**：品鉴行为日志，结构化存储
   - 格式：{时间, 品鉴者ID, 品鉴者级别, 对象(消息/文件/龙虾), 信号类型, 信号强度, 原始内容}
   - 存储位置：M3b（平台知识侧）
4. **输出**：定义品鉴记录的标准格式，供M2消费
   - M2根据品鉴记录更新MEMORY/Skill（学习）
   - 输出格式是M4和M2之间的契约

### 边界（不管什么）
- 不管品鉴信号**怎么影响龙虾学习**（M2的事 — M4只输出标准格式，M2决定怎么用）
- 不管品鉴者**有没有资格品鉴**（M0的事）
- 不管原始事件**怎么传过来**（M1的事）
- 不做品鉴信号的**价值判断**（"这个品鉴对不对"不是M4该管的）

### 前置依赖
```
M4 能工作的前提：
├── M1 必须推送reaction事件（当前❌缺失）
├── M0 必须区分品鉴者 vs 普通用户（当前❌缺失）
└── M3b 必须能存储品鉴行为日志（当前❌缺失）
```
**⚠️ M4的三个前置条件当前全部不满足。M4是最重要但当前完全无法启动的模块。**

### 归属
跨两侧（Octo侧采集传递，OpenClaw侧消费学习）

### 变更审批
黄楠（品鉴机制是核心产品逻辑）

### 当前状态
**几乎不存在**：
- ⚠️ emoji reaction在Octo侧已有存储，但webhook不推送
- ❌ 品鉴信号识别逻辑
- ❌ 品鉴行为日志
- ❌ 标准输出格式

## M5 · 协作协议（Collaboration Protocol）

### 定义
龙虾之间、龙虾与品鉴者之间的协作规则和组织结构。回答"龙虾之间怎么发现彼此、委托任务、形成组织"。

### 内部三层

**拓扑层（静态结构）**：
- Space管理（工作空间：多群的组织容器）
- Channel/群成员关系
- 群内龙虾列表与能力查询（channel_bots）

**协议层（动态交互）**：
- 握手协议（龙虾互相发现对方能力，Context Card交换）
- 任务委托（task_context透传、任务状态跟踪）
- 协作语义解析（task_context结构化、消息类型识别 — 原M1.5的协作语义部分）

**规范层（运行时规范，仅龙虾间）**：
- 信息分级（L1方向级/L2架构级/L3模块级/L4参数级）
- 品鉴请求格式（级别、主题、背景、选项、建议）
- 品鉴结果记录格式（D-XXX编号）
- 共识形成流程（讨论→多方输入→分歧判级→决策→广播）
- **约束**：只定义龙虾间的协作规范。只涉及一只龙虾自己的行为规范归M2。

### 接口
- 查询群内龙虾列表及能力（GET /groups/:id/members扩展版）
- 发送带task_context的消息（payload透传）
- 读写Context Card（依赖M3b）
- Space创建/查询

### 边界（不管什么）
- 不管消息怎么路由（M1的事）
- 不管龙虾自身怎么运行（M2的事）
- 不管品鉴信号怎么采集传递（M4的事）
- 不管具体业务逻辑（龙虾自己的事）
- **不定义单个龙虾的行为规范**（那归M2）

### 前置依赖
```
M5 能工作的前提：
├── M1 消息透传能力（payload自定义字段）
├── M0 身份查询（知道对方是谁）
└── M3b Context Card（知道对方能做什么）
```

### 归属
跨两侧

### 变更审批
黄楠 + 辉哥（协作协议影响全局）

### 当前状态
**早期**：
- ✅ Space概念已有（数据库有space_id字段，未充分使用）
- ✅ payload透传能力（task_context零改动可能即可用，待验证）
- ✅ 握手协议v1已完成（handshake-v1.md）
- ✅ 协作章程v0.2已完成（lobster-collaboration-charter-v0.2.md）
- ❌ 群内Bot列表+能力查询（靶子②，本周修）
- ❌ Context Card API

## OB · 可观测性（Observability）

### 定义
后置横切层。所有模块的数据消费者，提供审计追踪、系统指标、健康报告。回答"系统健不健康、知识在流动还是淤积"。

### 接口
- 审计追踪（所有知识变更可追溯 — 论文"可审计性"的工程保障）
- 系统指标（消息延迟、session用量、API错误率）
- 品鉴行为分析（品鉴频率、信号类型分布、品味生长轨迹）
- 组织健康报告（信息流动健康度、参与均匀度、决策闭环率）

### 边界（不管什么）
- 不做业务判断（只提供数据，不做决策）
- 不替代其他模块的日志（各模块自己记日志，OB做聚合分析）

### 设计约束
- **后置模块**：不阻塞其他模块的设计和实现
- **预留采集点**：每个模块在设计接口时，必须预留OB的数据采集点
- v1最小可用：品鉴行为计数 + 消息延迟监控

### 前置依赖
```
OB 能工作的前提：
├── M1 提供消息事件数据
├── M4 提供品鉴行为日志
├── M2 提供session运行数据
└── 各模块预留数据采集点
```

### 归属
跨两侧

### 变更审批
催化龙虾（彭特兰）主导，黄楠审批

### 当前状态
**不存在**：
- ❌ 自动化指标采集
- ⚠️ 人肉版健康报告（彭特兰每周手动评估）
- 计划：v1只做品鉴行为计数+消息延迟监控

### 收敛过程中的关键决策

| 决策 | 提出者 | 理由 |
|------|--------|------|
| M6(Octic)降级为外部接入源 | wuyun3 | Octic对M1来说就是另一个消息来源，不是系统内部模块 |
| 删除M1.5，平台语义归M1，协作语义归M5 | 彭特兰 | 区分"平台语义"和"协作语义"比独立模块更清晰 |
| M3拆为M3a+M3b | wuyun3 | 存储后端不同、访问权限不同、生命周期不同 |
| shared/归M3a，标注为M3b临时实现 | 三方共识 | 物理上在OpenClaw文件系统，但终态应迁移到平台侧 |
| IA升级为独立模块M0 | 毕达哥拉斯 | 品鉴者分级是核心业务逻辑，不是横切关注点 |
| M4加"输出"职责 | 毕达哥拉斯 | M4需定义标准格式供M2消费，否则记录了但没人用 |
| M5内部三层（拓扑+协议+规范） | 彭特兰 | 规范层承载L1-L4信息分级等协作规范 |
| M5规范层约束：仅龙虾间 | wuyun3 | 单龙虾行为规范归M2，防止M5变垃圾桶 |
| OB标注为后置模块 | 彭特兰 | 不阻塞其他模块，但各模块须预留数据采集点 |
| 每个模块需画前置依赖图 | wuyun3（采纳彭特兰M4依赖图） | 未来龙虾接需求时第一眼看依赖图 |
