# OCTO 协作操作系统 · 模块架构 v1.0

> 三方定稿：毕达哥拉斯（技术建模）× 彭特兰（组织信息流）× 无云（产品架构）
> 日期：2026-03-19
> 状态：定稿，待辉哥/娃爹品鉴确认

---

## 设计原则

1. **单一职责**：每个模块有清晰的一件事要做
2. **提供数据，不做判断**：平台层模块不替龙虾做业务决策（D-020）
3. **薄平台 + 厚Agent**：智能尽量在龙虾侧，平台做基础设施
4. **知识外化优于参数化**：所有知识在可读文本文件中，不在模型参数里

## M0 · 身份与权限（Identity & Access）

### 定义
回答"你是谁、什么级别、能做什么"。管理用户、龙虾、品鉴者的身份和权限。

### 边界（不管什么）
- 不管消息路由（归M1）
- 不管龙虾怎么运行（归M2）
- 不管品鉴信号怎么传递（归M4）

### 接口
- 查询用户身份（UID → 姓名/角色/品鉴者级别）
- 查询龙虾身份（Bot Token → 龙虾Profile/角色类型）
- 品鉴者分级管理（设置/查询品鉴者在不同领域的级别）
- Token生命周期管理

### 包含
- 用户身份管理（用户Profile + DMWork UID）
- 龙虾身份管理（Bot注册 + IDENTITY.md映射）
- 品鉴者分级（最高品鉴者/架构级/模块级，同一人不同领域可不同级别）
- 角色定义（品鉴者/专家龙虾/催化龙虾/执行龙虾/探索龙虾）
- Token管理

### 前置依赖
```
M0 无前置依赖（基础模块，被其他所有模块依赖）
```

### 变更审批人
辉哥（身份和权限是安全底线）

### 当前状态
**严重缺失**。Octo侧只有user，不区分品鉴者和普通用户。品鉴者分级完全不存在。

## M1 · 消息通道（Message Channel）

### 定义
消息/reaction/成员/状态事件的收发路由 + 平台语义增强。

### 边界（不管什么）
- 不管消息的业务语义（"这是指令还是反馈"归M5）
- 不管龙虾怎么处理消息（归M2）
- 不管品鉴信号的识别（归M4）

### 接口
- **入站**：webhook推送四类事件
  1. 消息事件（文本/图片/语音/文件）
  2. Reaction事件（emoji添加/移除）
  3. 成员变动事件（加入/退出/被踢）
  4. 在线状态事件
- **出站**：Bot API发送消息（含stream流式消息）
- **查询**：sync API查历史消息
- **平台语义增强**（原Plugin层职责）：
  - @门控（群里没@龙虾的消息静默缓存）
  - 身份解析（from_uid → 可读名字）
  - 历史上下文注入（被@时注入最近20条非@消息）
  - 出站处理（@名字反查为uid）

### 包含
- WuKongIM通信引擎（M1路由子能力）
- Channel管理（私聊/群聊）
- DMWork Plugin适配层（M1平台语义子能力）
- 消息历史存储与同步

### 前置依赖
```
M1 ← M0（身份解析需要M0提供UID→姓名映射）
```

### 变更审批人
研发侧（Octo团队）

### 当前状态
**运行中，有缺失**：
- ✅ 消息事件收发正常
- ❌ Reaction事件webhook不推送
- ❌ from_name/from_type字段缺失
- ❌ 成员变动事件不推送
- ❌ 大文件无保护（撑爆session）

## M2 · Agent 运行时（Agent Runtime）

### 定义
回答"龙虾怎么活——从注册到推理到学习"。管理龙虾的完整生命周期。

### 边界（不管什么）
- 不管消息怎么路由过来（归M1）
- 不管龙虾之间怎么协作（归M5）
- 不管知识存在哪里（归M3）
- 不管品鉴信号怎么采集（归M4）
- 只管涉及一只龙虾自身的行为规范；涉及两只以上龙虾交互的规范归M5

### 接口
- 注册Bot（BotFather）
- Dispatch消息到龙虾
- Session管理（创建/恢复/压缩/销毁）
- Context window组装规则（哪些永久层文件在什么条件下加载）
  - 例：MEMORY.md只在主session加载，群聊不加载（安全策略）
- 加载SOUL/MEMORY/Skill
- 工具调用、子agent调度

### 包含
- **BotFather**（Octo侧）：Bot注册、Token管理、命令菜单
- **Gateway**（OpenClaw侧）：Session管理、消息dispatch、context window组装
- **Adapter**（桥接）：DMWork ↔ OpenClaw协议转换
- **Compaction**：对话历史压缩

### 前置依赖
```
M2 ← M1（消息通过M1到达龙虾）
M2 ← M0（Bot注册需要M0的身份系统）
M2 ← M3a（运行时需要加载龙虾知识）
```

### 变更审批人
黄楠 + OpenClaw维护者

### 当前状态
**运行中，有缺失**：
- ✅ 基本消息收发和推理
- ❌ 自动session管理（大文件保护）
- ❌ Context window组装规则未显式定义

## M3a · 龙虾知识（Agent Knowledge）

### 定义
龙虾的文件知识系统，包含个体知识和共享知识库。OpenClaw文件系统侧。

### 边界（不管什么）
- 不管知识怎么创造（那是龙虾自己的事）
- 不管平台侧的知识（归M3b）
- 不管品鉴信号怎么采集（归M4，但M4的输出会更新M3a）

### 接口
- 读写知识文件（SOUL.md/MEMORY.md/memory/*.md/Skill文件）
- 语义搜索（memory_search）
- 全文搜索
- 版本历史查询（git）

### 包含
- **个体知识**：SOUL.md、MEMORY.md、memory/日志、Skill文件（归单个龙虾）
- **共享知识**：shared/目录、术语表、架构文档（归团队）
  - ⚠️ shared/是M3b的**临时降级实现**。依赖"龙虾恰好在同一台机器"这个物理巧合。M3b上线后应迁移。
- **知识检索**：memory_search语义搜索

### 前置依赖
```
M3a 无前置依赖（基础存储模块）
```

### 变更审批人
各龙虾自治（个体知识）/ 黄楠（共享知识库结构变更）

### 当前状态
**部分运行**：
- ✅ 个体知识读写正常
- ✅ 共享知识库（shared/）临时方案可用
- ❌ 跨机器龙虾无法共享（断裂点5）

## M3b · 平台知识（Platform Knowledge）

### 定义
Octo数据库侧的平台级知识：Context Card、用户Profile、共享知识（终态）。

### 边界（不管什么）
- 不管龙虾侧的文件知识（归M3a）
- 不管知识怎么被龙虾消费（归M2）

### 接口
- 读写Context Card（龙虾的公开身份名片）
- 读写用户Profile（品鉴者偏好、行为日志）
- 共享知识存储与检索（终态，替代shared/目录）

### 包含
- **Context Card**：龙虾的公开身份名片（能力描述、专长、可接受的任务类型）
- **用户Profile**：品鉴者偏好、个性化配置
- **品鉴行为日志**：M4输出的品鉴记录的持久化存储
- **共享知识服务**（终态）：跨龙虾的共享知识存储

### 前置依赖
```
M3b ← M0（用户Profile关联M0的身份系统）
```

### 变更审批人
Octo团队

### 当前状态
**基本不存在**：
- ❌ Context Card未实现
- ❌ 用户Profile未实现
- ❌ 共享知识服务未实现

## M4 · 品鉴信号（Connoisseur Signal）

### 定义
采集→传递→记录→输出。管理品鉴者反馈行为的全生命周期。

### 边界（不管什么）
- 不管品鉴者是谁、什么级别（归M0）
- 不管品鉴信号怎么影响龙虾学习（M4输出标准格式，M2消费它来调整行为）
- 不管品鉴请求怎么发起（归M5协议层）
- 只管品鉴记录的格式定义和输出

### 接口
- **采集**：从用户自然行为中识别品鉴信号
  - emoji reaction
  - 文字评价（正向/负向/修正）
  - 直接修改产出物
  - 修改龙虾的MEMORY/Skill
- **传递**：Octo侧把原始事件推给龙虾
- **记录**：品鉴行为日志（时间、对象、信号类型、强度）
- **输出**：定义品鉴记录的标准格式，供M2消费

### 品鉴信号强度谱系
emoji < 文字评价 < 直接修改产出物 < 修改Memory/Skill

### 前置依赖（⚠️ 三个前置条件当前全部不满足）
```
M4 能工作的前提：
├── M1 必须推送 reaction 事件（当前❌）
├── M0 必须区分品鉴者 vs 普通用户（当前❌）
└── M3b 必须能存储品鉴行为日志（当前❌）
```

### 变更审批人
黄楠（品鉴机制是核心产品逻辑）

### 当前状态
**最弱环节**：
- ❌ Reaction事件webhook不推送
- ❌ 品鉴者身份无区分
- ❌ 无品鉴行为日志
- ❌ 无标准输出格式
- 仅有的品鉴信号：龙虾自己在context window里观察到的文字反馈

## M5 · 协作协议（Collaboration Protocol）

### 定义
回答"龙虾之间怎么发现彼此、委托任务、形成组织"。仅管涉及两只以上龙虾交互的协作规范。

### 边界（不管什么）
- 不管单只龙虾自身的行为规范（归M2）
- 不管消息怎么路由（归M1）
- 不管品鉴信号怎么采集和记录（归M4）
- 判断标准：涉及一只龙虾自己→归M2，涉及两只以上龙虾交互→归M5

### 接口
- 查询群内龙虾列表（channel_bots）
- 发送带task_context的消息
- 读写Context Card
- 信息分级查询（L1-L4）

### 内部三层结构
1. **拓扑层**：Space/Channel/成员关系（静态结构）
   - 回答"组织长什么样"
2. **协议层**：握手/任务委托/Context Card交换/品鉴请求发起格式/共识形成流程
   - 回答"组织成员怎么交互"
3. **规范层**：信息分级L1-L4、信息共享规范、协作行为规范
   - 回答"交互应该遵循什么规则"
   - ⚠️ 约束：仅定义龙虾间协作规范，不管单龙虾自身行为

### 前置依赖
```
M5 ← M1（消息透传task_context需要M1支持）
M5 ← M0（协作中需要知道对方的角色和品鉴者级别）
M5 ← M3b（Context Card存储在M3b）
```

### 变更审批人
黄楠 + 辉哥（协作协议影响全局）

### 当前状态
**规划中**：
- ✅ 握手协议v1（handshake-v1.md）已完成
- ❌ task_context透传待验证
- ❌ Context Card未实现
- ❌ 群内龙虾列表API待确认
- ❌ Space功能已有但未充分使用

## OB · 可观测性（Observability）— 后置横切层

### 定义
回答"系统健不健康、知识在流动还是淤积"。横切所有模块。

### 特殊定位
**后置模块**——依赖其他模块先把数据采集能力建好。不阻塞其他模块设计，但每个模块在设计接口时必须预留OB的数据采集点。

### 接口
- 审计追踪（所有知识变更可追溯）
- 系统健康指标（session用量、消息延迟、API错误率）
- 品鉴行为分析（品味生长的可量化观测）
- 信息流动健康报告

### v1最小可用
- 品鉴行为计数
- 消息延迟监控

### 变更审批人
黄楠

### 当前状态
**基本不存在**，健康报告只能靠人肉版。

## 核心数据流（一条消息的完整生命周期）

```
用户消息
  → M1 路由 → M1 平台语义增强（@解析、身份、历史注入）
  → M0 鉴权（谁发的、什么级别）
  → M2 dispatch到龙虾 → M3a 加载知识 → LLM推理 → 回复
  → M1 出站发出
  → 品鉴者反馈
  → M1 路由 reaction事件
  → M4 采集品鉴信号 → M4 记录 → M4 输出标准格式
  → M2 消费品鉴记录 → M3a 更新知识（学习）
  → 龙虾间协作
  → M5 协议 → M3b 查Context Card → M1 透传task_context
```
