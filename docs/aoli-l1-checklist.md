# L1 一审 Checklist（AoLi 维护）

> 核心问题：**对用户有没有价值，方案逻辑通不通。**
> 不管架构影响、技术可行性（那是二审的事）。

## 串行漏斗（前一关没过后面不用看）

### 关卡 -1：档位判断（最先做）

| 档位 | 判断条件 | 我写什么 | 我自检什么 | 篇幅 |
|------|---------|---------|-----------|------|
| **简单** | 单端、单场景、无新交互（改文案/删按钮/调样式/加入口） | 一句话摘要 + 变更列表 + 验收标准 | 只过关卡 0（规范检查） | ≤ 20 行 |
| **中等** | 有新交互流程，或多端/多角色，模块内闭环 | 一句话摘要 + 痛点验证 + 交互流程 + 关键边界 + 验收标准 | 关卡 0-4 全走，每章节精简 | ≤ 50 行 |
| **复杂** | 跨模块依赖、引入新概念、不可逆变更、需分阶段 | 一句话摘要 + 完整 PRD + 关联需求 + 分阶段计划 | 关卡 0-4 全走 | 按需 |

**仲裁规则：档位就高不就低。** 我判简单但 JOJO 判中等 → 按中等写。
**L1 自检记录是工作底稿，任何档位都不写进 Issue 正文。**

### 关卡 0：PRD 规范检查（前置红线）

> 命中任一条 → 先改再提审，不进后续关卡。

- [ ] PRD 正文中有没有代码块（```json、```yaml 等）？→ 删掉，用自然语言描述
- [ ] 有没有定义 API 参数名、字段结构、数据类型？→ 改成描述用户需要传递/看到的信息
- [ ] 有没有出现内部实现组件名（Gateway、Adapter、Handler、Dispatcher 等）？→ 改成用户操作的产品功能名
- [ ] 有没有画系统架构图或时序图？→ 只保留用户操作流程图或页面原型
- [ ] 验收标准是从技术视角还是用户视角写的？→ 改成用户可感知的期望行为
- [ ] 有没有给出具体性能数值（QPS、延迟 ms、50KB 等）？→ 用产品语言：「3 秒内看到发送成功」

**自检口诀：写完每段话问自己——「这句话换成研发来写更合适吗？」如果是，删掉。**

### 关卡 1：痛点验证（俞军）

- [ ] 有真实的用户证据吗？（谁提的、什么场景、能复现吗）
- [ ] 用户价值 > 0？新体验 - 旧体验 - 替换成本为正？
- [ ] 不做会怎样？有没有现成替代方案？
- [ ] 影响面多大？1 个人还是一类用户？
- [ ] 这是「必须有」还是「有了更好」？

→ 没有用户证据或痛点不成立 → `❌ 打回`

### 关卡 2：问题定义（Don Norman）

- [ ] 用户说的方案和底层问题对得上吗？（有没有 XY Problem）
- [ ] 有没有更简单的解法？（文案/引导/文档能解决就不改代码）
- [ ] 需求边界清晰吗？做到什么程度算完？
- [ ] 交互符合直觉吗？不需要教程就能用？

→ 问题定义偏了 → `🔄 打回，要求重新定义问题`

### 关卡 3：AI 友好性（Patrick Collison）

- [ ] AI Agent 通过 API/webhook 交互，设计对它合理吗？
- [ ] 人类界面和 API 有没有信息不对称？（人能看到但 Bot 拿不到）
- [ ] Bot 在这个场景下能正常工作吗？
- [ ] 符合「AI 是一等公民」的北极星吗？

→ 只考虑了人类忽略了 AI → `⚠️ 需补充 AI 视角`

### 关卡 4：边界与异常

- [ ] 空状态、错误状态、极端情况覆盖了吗？
- [ ] 权限边界清楚吗？（谁能做、谁不能做）
- [ ] 跟现有功能有冲突吗？

→ 边界没想清楚 → `⚠️ 需补充边界`

## 自检通过后

1. 在 PM Issue 正文中包含完整 PRD（不只是一句话描述）
2. 打标签：`待审核` + 优先级（`P0` / `P1` / `P2`）
3. 关闭对应的 pool Issue，评论写「已转 dmwork-PM #xxx」

## 一审不看的

技术可行性、工时估算、架构影响、模块边界、跨模块连锁反应 — 这些是 JOJO 二审的事。
