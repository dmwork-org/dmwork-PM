# L1 一审 Checklist（AoLi 维护）

> 核心问题：**对用户有没有价值，方案逻辑通不通。**
> 不管架构影响、技术可行性（那是二审的事）。

## ⚠️ 反模板化原则（2026-03-31 佳佳批评后新增）

**关卡是思考工具，不是 PRD 章节模板。**

- 关卡在脑子里过，不要逐关卡往 Issue 里填
- PRD 的结构跟着需求走，不跟着模板走
- 问自己：**这个需求，品鉴者判断「做不做」最需要知道什么？** 把那个写在最显眼的位置
- 不需要的章节不写。痛点一句话说清的，不要凑一个「痛点验证」章节
- 每个 PRD 应该长得不一样，如果 15 个 Issue 章节结构一模一样，说明在套模板

**反面例子：**
- #185 隐私风险是核心争议点，但被埋在「关键边界」第五个 bullet 里
- #143 痛点一目了然，却还写了一段「痛点验证」凑格式
- 15 个 Issue 全是「一句话→痛点验证→交互流程→关键边界→验收标准」，cookie-cutter

**正面做法：**
- #185 应该开头就写清楚隐私风险和管控方案——那是佳佳判断做不做的关键
- #143 一句话+交互流程+验收标准就够了，不需要「痛点验证」章节
- 结构因需求而异，重要信息前置

## 串行漏斗（前一关没过后面不用看）

### 关卡 -1：档位判断（最先做）

| 档位 | 判断条件 | 篇幅 |
|------|---------|------|
| **简单** | 单端、单场景、无新交互 | ≤ 20 行 |
| **中等** | 有新交互流程，或多端/多角色 | ≤ 50 行 |
| **复杂** | 跨模块依赖、引入新概念、不可逆 | 按需 |

**仲裁规则：档位就高不就低。**
**L1 自检记录是工作底稿，不写进 Issue 正文。**

### 关卡 -0.5：架构预审触发判断（2026-03-31 三方共识）

> 在档位判断之后、写方案之前做。命中任一条 → 先 @ Brooks 做架构预审（5分钟方向建议），拿到方向再写 PRD。

- [ ] 涉及 M4（品鉴信号）或 M5（协作协议）等**当前空白模块**？→ 触发预审
- [ ] 需要**新建数据表**？→ 触发预审
- [ ] 需要**新建 API 端点**？→ 触发预审

**不触发的情况：** 已有模块（M0/M1/M2/M3a/M3b/OB）内的功能增强，直接写方案。
**预审地点：** Octo 群 @ Brooks，只要方向建议，不要详细设计。

### 关卡 0：PRD 规范检查（前置红线）

> 命中任一条 → 先改再提审，不进后续关卡。

- [ ] PRD 正文中有没有代码块（```json、```yaml 等）？→ 删掉，用自然语言描述
- [ ] 有没有定义 API 参数名、字段结构、数据类型？→ 改成描述用户需要传递/看到的信息
- [ ] 有没有出现内部实现组件名（Gateway、Adapter、Handler、Dispatcher 等）？→ 改成用户操作的产品功能名
- [ ] 有没有画系统架构图或时序图？→ 只保留用户操作流程图或页面原型
- [ ] 验收标准是从技术视角还是用户视角写的？→ 改成用户可感知的期望行为
- [ ] 有没有给出具体性能数值（QPS、延迟 ms、50KB 等）？→ 用产品语言：「3 秒内看到发送成功」

**自检口诀：写完每段话问自己——「这句话换成研发来写更合适吗？」如果是，删掉。**

### 关卡 1：痛点验证（俞军）

- [ ] 有真实的用户证据吗？（谁提的、什么场景、能复现吗）
- [ ] 用户价值 > 0？新体验 - 旧体验 - 替换成本为正？
- [ ] 不做会怎样？有没有现成替代方案？
- [ ] 影响面多大？1 个人还是一类用户？
- [ ] 这是「必须有」还是「有了更好」？

→ 没有用户证据或痛点不成立 → `❌ 打回`

### 关卡 2：问题定义（Don Norman）

- [ ] 用户说的方案和底层问题对得上吗？（有没有 XY Problem）
- [ ] 有没有更简单的解法？（文案/引导/文档能解决就不改代码）
- [ ] 需求边界清晰吗？做到什么程度算完？
- [ ] 交互符合直觉吗？不需要教程就能用？

→ 问题定义偏了 → `🔄 打回，要求重新定义问题`

### 关卡 3：AI 友好性（Patrick Collison）

- [ ] **入口对等检查**：这个功能的所有入口，Bot 和人类都能触达吗？
  - 如果不能 → **列出 Bot 侧的能力缺口**（具体哪个入口 Bot 走不通）
- [ ] 人类界面和 API 有没有信息不对称？（人能看到但 Bot 拿不到）
  - 如果有 → **列出信息差异项**（哪些字段/状态人类能看到但 Bot API 没暴露）
- [ ] Bot 在这个场景下能正常完成完整流程吗？（不是"能用"，是"能用完"）
- [ ] 符合「AI 是一等公民」的北极星吗？

→ 只考虑了人类忽略了 AI → `⚠️ 需补充 AI 视角，附 Bot 侧能力缺口清单`

### 关卡 4：边界与异常

- [ ] 空状态、错误状态、极端情况覆盖了吗？
- [ ] 权限边界清楚吗？（谁能做、谁不能做）
- [ ] 跟现有功能有冲突吗？

→ 边界没想清楚 → `⚠️ 需补充边界`

## 自检通过后

1. 在 PM Issue 正文中包含完整 PRD（不只是一句话描述）
2. 打标签：`待审核` + 优先级（`P0` / `P1` / `P2`）
3. 关闭对应的 pool Issue，评论写「已转 dmwork-PM #xxx」

## 一审不看的

技术可行性、工时估算、架构影响、模块边界、跨模块连锁反应 — 这些是 JOJO 二审的事。
