# PRD 写作标准

> 产品说「要什么」，研发定「怎么做」。

## 一、PRD 应该写什么

### 1. 问题分析
- **痛点验证**：谁、在什么场景下、遇到什么问题
- **真实案例**：至少一个具体的用户故事或反馈原文
- **影响范围**：多少用户受影响、频率如何
- **现有替代方案**：用户现在怎么绕过这个问题
- **必须有 vs 有了更好**：判断优先级依据

### 2. 用户故事
- 格式：「作为一个 ___，我希望 ___，以便 ___」
- 覆盖主要角色（用户 / AI / 管理员等）

### 3. 交互流程（用户视角）
- 用户操作 → 看到什么 → 得到什么反馈
- 用自然语言描述，不画时序图或系统架构图
- 配图只展示用户界面（页面原型、流程图），不展示内部模块

### 4. 边界条件（用户可感知的）
- 正常态：一切顺利时用户看到什么
- 异常态：出错了用户看到什么提示
- 极端情况：内容过长、网络断开、权限不足等场景下的用户体验

### 5. 产品约束
- 用户可感知的限制条件（如「必须支持离线场景」「必须兼容移动端」）
- 写约束，不写方案。研发根据约束自行选择技术方案

### 6. 验收标准
- 从用户可感知的角度写
- 描述期望行为，不描述实现方式

## 二、PRD 不该写什么

| ❌ 不该出现的内容 | ✅ 应该怎么写 |
|---|---|
| API 接口定义（参数名、字段结构） | 描述用户需要传递/看到的信息 |
| JSON / YAML 代码块 | 用自然语言描述数据内容 |
| 数据库表结构、存储方案 | 描述用户期望的数据持久化行为 |
| 技术协议（file://、WebSocket 等） | 描述用户期望的功能效果 |
| 内部模块名（Gateway、Adapter 等） | 描述用户操作的产品功能名 |
| 性能指标具体数值（QPS、延迟 ms） | 用产品语言：「3 秒内看到发送成功」 |
| 系统架构图、时序图 | 画用户操作流程图或页面原型 |

## 三、自检方法

### 一句话嗅探法
写完每段话问自己：**「这句话换成研发来写更合适吗？」** 如果是，删掉。

### 代码块红线
PRD 里出现代码块（```json、```yaml 等）= 越界。没有例外。

### 用户感知原则
**用户能感知到的，写；用户感知不到的内部逻辑，不写。**

比如：
- ✅「发送过长的背景信息时，系统应提示用户精简内容」（用户能感知）
- ❌「context 超过 50KB 时截断并警告」（技术约束，用户不关心 50KB）
- ✅「用户下次打开会话时，之前委托的背景信息仍然可见」（用户能感知）
- ❌「context 随消息一起存入 session history」（内部实现）

### 期望行为替代实现方式
| ❌ 实现方式 | ✅ 期望行为 |
|---|---|
| Gateway 序列化 context 随 message 传递 | 接收方打开任务时能直接看到完整背景 |
| context 作为 system message 注入 | 接收方无需追问即可理解任务 |
| file:// 引用不可用时标记为 [unavailable] | 引用的文件无法访问时，告知用户并不阻断发送 |

## 四、PRD 模板

```markdown
# ✨ [模块] 功能名称

## 一、问题分析
### 痛点验证
### 真实案例
### 影响范围
### 现有替代方案

## 二、用户故事

## 三、方案设计
### 交互流程（用户视角）
### 边界条件

## 四、产品约束

## 五、验收标准

## 六、关联需求
```

## 五、评审检查项（JOJO 用）

评审时逐条检查，命中任一条即打回：
- [ ] PRD 中是否出现代码块？
- [ ] PRD 中是否定义了 API 参数名、字段结构、数据类型？
- [ ] PRD 中是否出现内部模块名（Gateway、Adapter、Webhook 等）？
- [ ] PRD 中是否画了系统架构图或时序图？
- [ ] 验收标准是否从技术视角而非用户视角写的？
- [ ] 是否给出了具体的性能数值（而非产品语言的体验描述）？
