# ADR-001: 双层架构——业务层与通讯层分离

> 将 IM 业务逻辑（dmworkim）与底层消息传输（WuKongIM）分离，通过 gRPC/HTTP 交互。

## 概述

这是 Octo/DMWork 最核心的架构决策：不自研 IM 引擎，而是基于 WuKongIM 构建业务层。

---

## 状态

**已接受（Accepted）**

---

## 背景与问题

构建一个即时通讯平台，需要解决两类截然不同的问题：

**通讯层问题**（技术复杂，已有成熟方案）：
- WebSocket 长连接管理（大量并发连接）
- 消息路由与可靠投递（SEND/SENDACK/RECV/RECVACK 协议）
- 频道订阅管理（谁在哪个频道）
- 消息持久化（海量 append-only 写入）
- 在线状态追踪

**业务层问题**（业务专属，需要定制）：
- 用户身份、好友关系、权限验证
- 群组生命周期管理
- AI Bot 接入（BotFather 系统）
- Space 多租户隔离
- 消息内容业务处理（敏感词、撤回、编辑）
- 推送通知（离线消息）
- 文件存储

如果把两类问题混在一个系统里，会导致：
- 通讯引擎和业务逻辑相互污染，难以独立演进
- 重复造轮子（高性能 WebSocket 引擎并非容易之事）
- 测试困难（业务逻辑测试需要真实的 WebSocket 环境）

---

## 决策

**采用双层架构**：

```
┌─────────────────────────────────────────────────────┐
│           DMWorkim（业务逻辑层）                       │
│  用户管理 · 群组管理 · Bot系统 · 工作台 · 推送 · 文件  │
└─────────────────────────┬───────────────────────────┘
                          │ HTTP API / gRPC Webhook
┌─────────────────────────▼───────────────────────────┐
│           WuKongIM（底层通讯层）                       │
│    WebSocket 长连接 · 消息存储 · 频道管理 · 路由       │
└─────────────────────────────────────────────────────┘
```

**选择 WuKongIM 作为通讯层**，理由：
- 开源（GitHub），已验证的高性能 IM 引擎
- 支持 gRPC Webhook，可以向业务层回调消息事件
- 支持自定义业务数据源（`WK_DATASOURCE_ADDR`），让 WuKongIM 调用 dmworkim 做好友/权限校验
- 自定义二进制协议，高效传输

**两层交互接口**：

| 方向 | 协议 | 用途 |
|------|------|------|
| dmworkim → WuKongIM | gRPC | 发消息（IMSendMessage）、管理频道（IMCreateOrUpdateChannel）、控制流式消息 |
| dmworkim → WuKongIM | HTTP | 管理 API（白名单/黑名单/在线状态查询） |
| WuKongIM → dmworkim | gRPC Webhook | 消息事件回调（robotMessageListen）、好友/权限校验 |

---

## 理由

### 优势

1. **专注分工**：WuKongIM 专注高性能传输，dmworkim 专注业务语义。各自可以独立优化、独立发布。

2. **开源杠杆**：WuKongIM 是成熟开源项目，不需要从零实现高性能 WebSocket 引擎。节省大量研发时间。

3. **可替换性**：如果将来 WuKongIM 无法满足需求，可以替换为其他通讯引擎，业务层代码影响最小（只需更换 gRPC/HTTP 客户端）。

4. **测试友好**：业务层单元测试可以 mock WuKongIM 接口，不需要真实 IM 连接。

5. **技术债务隔离**：自定义二进制协议的复杂性（DH+AES、粘包处理等）封装在 WuKongIM 和 dmwork-adapters 中，业务层不感知。

### 劣势与权衡

1. **运维复杂度增加**：需要同时运维两个服务（dmworkim + WuKongIM），增加 Docker Compose 复杂度。
   - 权衡：接受。Docker Compose 已将复杂度封装，一键启动。

2. **跨进程调用延迟**：dmworkim 通过 gRPC 调用 WuKongIM 比进程内调用多一次网络往返。
   - 权衡：接受。同一台服务器上的 gRPC 延迟可忽略不计（<1ms），IM 场景不是超低延迟要求。

3. **WuKongIM 版本耦合**：需要保持 dmworkim 与 WuKongIM API 版本兼容。
   - 权衡：接受。WuKongIM API 相对稳定，接口通过 gRPC proto 定义，版本变化可感知。

---

## 后果

**正向影响**：
- 架构清晰，新开发者能快速理解"业务在 dmworkim，传输在 WuKongIM"
- WuKongIM 上游更新可以直接利用（性能提升、bug 修复）
- 业务层 18 个模块可以独立迭代，不影响通讯层

**需要注意**：
- 所有"发消息"操作都需要通过 gRPC 调用 WuKongIM，不能直接写数据库
- WuKongIM 的好友/权限校验会回调 dmworkim，需要 dmworkim 暴露校验接口
- 消息存储在 WuKongIM 侧，dmworkim 需要通过 API 同步历史消息

---

## 替代方案（被否决）

### 方案 A：全自研 IM 引擎
- **否决理由**：开发周期长（WebSocket 引擎、消息路由、持久化至少 6 个月），且高性能 IM 引擎是专业领域，团队不擅长。

### 方案 B：使用商业 IM SDK（如环信、融云）
- **否决理由**：数据不在自己手里，私有化部署成本高，定制化能力受限（无法支持 Bot 流式消息等定制特性）。

### 方案 C：单体架构（业务+通讯合并）
- **否决理由**：代码耦合度高，通讯引擎的性能优化（C 风格代码、内存管理）与 Go 业务代码混合，难以维护。
