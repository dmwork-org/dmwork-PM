## DMWorkim 服务问题

### 服务启动后 API 无响应

**症状**：HTTP 请求到 `:8080` 超时或连接被拒绝。

**排查步骤**：

1. 检查容器是否运行：
   ```bash
   docker compose ps
   docker compose logs dmworkim --tail=50
   ```

2. 检查端口绑定：
   ```bash
   curl http://localhost:8080/v1/health
   ```

3. 常见原因：
   - MySQL 未就绪（dmworkim 启动时 MySQL 尚未接受连接）
   - 迁移失败（SQL 错误）
   - 配置错误（YAML 解析失败）

4. MySQL 未就绪时，dmworkim 会在日志中打印连接失败并可能退出。确保启动顺序：
   ```bash
   docker compose up -d mysql redis
   sleep 10
   docker compose up -d dmworkim wukongim
   ```

---

### 数据库迁移失败

**症状**：dmworkim 启动日志中出现 `migration failed` 或 `sql: duplicate column`。

**排查**：
```bash
docker compose logs dmworkim | grep -i migration
```

**常见原因**：
- 手动修改过数据库表结构，与迁移文件冲突
- MySQL 版本不兼容

**解决**：
- 查看具体失败的 SQL 文件（日志中会显示文件名）
- 手动修复数据库状态后重启

---

### MODE=debug 的安全隐患

> ⚠️ **重要**：`configs/tsdd.yaml` 中未配置 `mode` 时，`ConfigureWithViper()` 默认使用 `DebugMode`（"debug"），而非 "release"。

**症状**：生产环境运行在 debug 模式，日志过多，可能暴露内部信息。

**解决**：在配置文件或环境变量中明确设置：
```yaml
# configs/tsdd.yaml
mode: "release"
```
或：
```bash
MODE=release docker compose up -d
```

## WuKongIM 连接问题

### 客户端无法建立 WebSocket 连接

**症状**：客户端（iOS/Android/Web）无法连接到 `ws://ip:5200`。

**排查**：

1. 检查 WuKongIM 是否运行：
   ```bash
   docker compose logs wukongim --tail=30
   curl http://localhost:5001/v1/health
   ```

2. 检查防火墙/安全组：端口 5200 是否对外开放。

3. 检查 Webhook 配置：WuKongIM 必须能访问 dmworkim 的 Webhook 地址：
   ```bash
   # WuKongIM 会调用这个地址
   curl http://dmworkim:8080/v1/datasource
   ```

---

### CONNACK 错误码

WebSocket 连接握手完成后，WuKongIM 返回 CONNACK 包，reasonCode 含义：

| reasonCode | 含义 | 处理方式 |
|-----------|------|---------|
| 1 | 连接成功 | — |
| 0 | 被踢下线（Kicked） | 不重连，通知用户 |
| 其他 | 连接失败 | 可重连 |

---

### Bot 离线 / 无法收到消息

**症状**：Bot 注册成功但收不到消息事件。

**排查步骤**：

1. 确认 im_token 有效：
   ```bash
   # 重新注册 Bot，获取新的 im_token
   curl -X POST https://api.example.com/v1/bot/register \
     -H "Content-Type: application/json" \
     -d '{"bot_token": "bf_xxx"}'
   ```

2. 确认 WuKongIM WebSocket 连接建立（查看 Bot 日志）。

3. 确认 Bot 心跳正常发送：
   ```bash
   # Bot 应每 60 秒发送一次心跳
   POST /v1/bot/heartbeat
   ```

4. 如果使用长轮询事件接口（`/v1/bot/events`），检查请求是否正常循环。

---

### Bot 消息重复接收

**症状**：同一条消息被 Bot 处理多次。

**原因**：事件未正确 ACK，下次轮询时 WuKongIM 重新投递。

**解决**：
- 确认每次处理完事件后调用 `POST /v1/bot/events/:event_id/ack`
- 检查幂等处理逻辑（用 `client_msg_no` 去重）

## OpenClaw 问题

### OpenClaw Gateway 无法连接

**症状**：`openclaw gateway status` 显示未运行。

**排查**：
```bash
# 检查服务状态
openclaw gateway status

# 查看日志
openclaw logs --follow

# 健康检查
openclaw health

# 重启
openclaw gateway restart
```

### Bot Token 过期 / 无效

**症状**：Bot API 返回 401 Unauthorized。

**排查**：
- Bot Token 是 `bot_token` 字段（`bf_` 前缀），在机器人创建后生成
- 通过 `/v1/user/bots/:bot_id/token` 获取或重新生成
- 管理员可通过 `POST /v1/manager/robots/:robot_id/revoke_token` 重置 Token

---

### WebSocket 频繁断线重连

**症状**：客户端日志频繁出现断线重连（dmwork-adapters 中 3 秒后重连）。

**可能原因**：

| 原因 | 症状特征 | 解决 |
|------|---------|------|
| 心跳超时 | 60 秒内未收到 PONG（最多 3 次）| 检查网络稳定性 |
| NAT 超时 | 定时断线，间隔规律 | 调小心跳间隔 |
| 服务端重启 | 所有客户端同时断线 | 检查 WuKongIM 服务状态 |
| Token 过期 | 断线后重连失败（CONNACK reasonCode≠1）| 刷新 im_token |

---

### AES 密钥协商失败

**症状**：WebSocket 握手后连接立即断开，无法解密消息。

**背景**：WuKongIM 使用 DH 密钥交换 + AES 加密（密钥派生：sharedKey → base64 → MD5 → 前16字符）。

**排查**：
- 检查 CONNACK 包中 `serverKey` 和 `salt` 字段是否正确接收
- AES IV 派生：`salt.length > 16 ? salt[0:16] : salt`（完整 salt 作为 IV）
- 确认 AES 密钥派生步骤正确

## 日志分析

### dmworkim 关键日志位置

```bash
# 容器日志
docker compose logs dmworkim

# 按关键词过滤
docker compose logs dmworkim | grep -E "ERROR|WARN|migration"

# 实时跟踪
docker compose logs -f dmworkim
```

### 常用诊断命令

```bash
# 检查数据库连接
docker compose exec dmworkim wget -q -O- http://localhost:8080/v1/health

# 检查 Redis
docker compose exec redis redis-cli ping

# 检查 MySQL
docker compose exec mysql mysql -u root -p dmwork -e "SHOW TABLES;"

# 检查 WuKongIM
curl http://localhost:5001/v1/health
```
