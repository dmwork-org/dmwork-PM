# 消息分片设计

[[概览|← 返回概览]]

DMWorkim 的消息存储采用**水平分片（Horizontal Sharding）**设计，将消息按 `message_id` 哈希分散到多张表中，减少单表压力。

## 分片策略

### webhook 模块（消息正文分片）

WuKongIM 通过 Webhook（gRPC）将消息推送到 DMWorkim 存储。消息按 `message_id % 5` 路由到不同分片表：

```
message_id % 5 = 0 → message   表
message_id % 5 = 1 → message1  表
message_id % 5 = 2 → message2  表
message_id % 5 = 3 → message3  表
message_id % 5 = 4 → message4  表
```

**5 张表结构完全相同**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | PK AUTO_INCREMENT |
| message_id | VARCHAR(20) | 消息唯一 ID（全局，WuKongIM 分配） |
| message_seq | bigint | 消息序列号（频道内严格递增） |
| client_msg_no | VARCHAR(40) | 客户端消息编号（去重） |
| channel_id | VARCHAR(100) | 频道 ID |
| channel_type | smallint | 频道类型 |
| from_uid | VARCHAR(40) | 发送者 uid |
| type | integer | 消息类型（ContentType） |
| payload | blob | 消息体（加密后的二进制） |
| timestamp | integer | 发送时间戳（Unix 秒） |
| created_at | timestamp | 记录创建时间 |

### message_user_extra 分片（用户消息扩展）

用户对消息的个人扩展（单向删除、语音已读）也采用分片：

```
uid hash % 3 → message_user_extra / message_user_extra1 / message_user_extra2
```

| 字段 | 说明 |
|------|------|
| uid | 用户 UID |
| message_id | 消息 ID |
| voice_readed | 语音消息是否已读（0/1） |
| message_is_deleted | 该用户视角下消息是否已删除（0/1） |

### channel_offset 分片（频道读取偏移）

记录每个用户在每个频道的消息读取位置：

```
uid hash % 3 → channel_offset / channel_offset1 / channel_offset2
```

| 字段 | 说明 |
|------|------|
| uid | 用户 UID |
| channel_id | 频道 ID |
| channel_type | 频道类型 |
| message_seq | 已读到的最大消息序列号 |

## 消息扩展（message_extra）

`message_extra` 表**不分片**，通过 `message_id` 唯一索引关联到分片表中的消息：

| 字段 | 说明 |
|------|------|
| message_id | 消息唯一 ID |
| revoke | 是否撤回（0/1） |
| revoker | 撤回者 uid |
| readed_count | 已读用户数 |
| content_edit | 编辑后的正文（TEXT） |
| content_edit_hash | 编辑内容 hash |
| edited_at | 编辑时间戳（Unix 秒） |
| is_pinned | 是否置顶（0/1） |
| is_deleted | 是否删除（0/1） |

## 消息 Reaction（reaction_users）

消息 emoji 回应表，**不分片**：

```sql
reaction_users (
  message_id,   -- 关联消息
  channel_id,   -- 频道
  channel_type,
  uid,          -- 回应的用户
  emoji,        -- 回应的 emoji
  is_deleted,   -- 是否已取消
  seq           -- 递增序号（同步用）
)
```

## 频道偏移追踪

消息读取进度通过多个维度追踪：

```
channel_offset[0~2]    ← 用户在频道的当前读取位置
  └── uid + channel_id + channel_type → message_seq

device_offset          ← 每个设备独立的读取位置（多设备场景）
  └── uid + device_uuid + channel_id + channel_type → message_seq

user_last_offset       ← 用户在频道的最新读取位置（聚合）
  └── uid + channel_id + channel_type → message_seq

conversation_extra.browse_to  ← 影响未读数计算的最大 message_seq
```

## 序列号（message_seq）机制

`message_seq` 是每个频道内**严格单调递增**的序列号，由 WuKongIM 分配：

- 同一频道内 message_seq 唯一且连续
- 客户端通过 `message_seq` 实现增量同步
- `channel_offset.message_seq` 记录用户已同步到的位置

## 数据流

```
用户发消息
  ↓
WuKongIM（底层通讯层）
  ├── 分配 message_id + message_seq
  ├── 存储到自己的存储层
  └── 通过 gRPC Webhook 推送给 DMWorkim
       ↓
DMWorkim webhook 模块
  ├── 计算 message_id % 5
  └── 写入对应分片表（message / message1~4）
```
