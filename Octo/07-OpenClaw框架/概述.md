# OpenClaw 框架概述

> **"The product is the assistant, Gateway is just the control plane."**
>
> 这是 OpenClaw 最核心的设计哲学。Gateway 不是产品，只是基础设施控制面。真正的产品是助手本身——它的人格（SOUL.md）、它的记忆（MEMORY.md）、它的工作方式（AGENTS.md）。

## 什么是 OpenClaw

OpenClaw 是一个**个人 AI 助手框架**——你自己运行、自己掌控的 AI 助手，可以在你已经使用的 IM 平台上回答你的问题。

| 属性 | 值 |
|------|---|
| npm 包名 | `openclaw` |
| 当前版本 | `2026.3.13` |
| 描述 | Multi-channel AI gateway with extensible messaging integrations |
| 安装位置 | `/opt/homebrew/lib/node_modules/openclaw` |
| 包结构 | `dist/`（编译产物）, `extensions/`, `skills/`（**无 src/**） |

> ⚠️ **注意**：npm 包不包含 TypeScript 源码（`src/`），只有编译后的 `dist/` 目录。源码路径仅在开发者仓库中存在。

## 产品定位

**你的私有 AI 助手，多渠道触达，你的设备，你的数据。**

### 支持的渠道（20+）

WhatsApp、Telegram、Slack、Discord、Google Chat、Signal、iMessage、BlueBubbles、IRC、Microsoft Teams、Matrix、飞书(Feishu)、LINE、Mattermost、Nextcloud Talk、Nostr、Synology Chat、Tlon、Twitch、Zalo、Zalo Personal、WebChat

### 扩展能力

- **语音**：macOS/iOS/Android 上的语音唤醒（Wake Word）和对话模式（Talk Mode）
- **Live Canvas**：Agent 驱动的可视化工作区（A2UI）
- **多端 Companion App**：macOS 菜单栏、iOS/Android Node

## 设计哲学

```
核心定位：
  Gateway = 控制面（基础设施）
  助手 = 产品（身份、记忆、行为）
```

这个理念导出了四个关键决策：

| 决策 | 说明 |
|------|------|
| 单一 Gateway 进程 | 一个进程控制所有消息渠道，避免分布式复杂度 |
| 本地优先（Local-First） | Gateway 跑在你自己的设备上，数据不过第三方服务器 |
| Agent Workspace 是一等公民 | SOUL.md、AGENTS.md、MEMORY.md 定义助手的身份和行为 |
| Channel 是可插拔管道 | 20+ IM 平台通过插件机制接入，核心逻辑与渠道解耦 |

## 关键设计模式

### 1. 插件化（Plugin-First）
所有渠道、Provider 都是插件。核心只定义接口，具体实现在 `extensions/` 中。

### 2. Session Key 路由
消息通过结构化 session key 路由，格式编码了：agentId、channel、peer kind、peer id。
例：`agent:xijing:telegram:group:1234567890`

### 3. Workspace Contract（工作区契约）
`SOUL.md + AGENTS.md + USER.md` 三件套是 Agent 身份的"宪法"，每个 session 启动时注入到 system prompt。

### 4. 多 Agent 隔离
每个 Agent 有独立工作区、session store、auth profiles。通过 bindings 将不同渠道/用户路由到不同 Agent。

### 5. 热重载（Hybrid Mode）
默认 `hybrid` 模式：热安全变更直接应用，需要重启的变更自动重启 Gateway。
