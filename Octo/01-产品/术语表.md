# 术语表

> Octo + OpenClaw 生态全术语速查手册，覆盖 IM 核心、Bot 系统、加密协议、OpenClaw 框架等所有领域。

## 概述

本文档汇总了 Octo/DMWork 和 OpenClaw 生态系统中所有核心术语，按领域分类整理。

---

## IM 核心术语

### WuKongIM（悟空IM）

**定义**：Octo 底层通讯引擎，开源 IM 基础设施。

- 负责 WebSocket 长连接管理、消息路由与投递、频道订阅、消息持久化
- 与业务层（dmworkim）通过 **gRPC**（接收业务指令）和 **HTTP Webhook**（推送消息事件）交互
- 使用自定义二进制协议（帧格式：`[PacketType(4bit)|Flags(4bit)][VariableLength][Body]`）
- 支持包类型：CONNECT(1)、CONNACK(2)、SEND(3)、SENDACK(4)、RECV(5)、RECVACK(6)、PING(7)、PONG(8)、DISCONNECT(9)

---

### Channel（频道）

**定义**：WuKongIM 中消息路由的基本单位，由 `channel_id` + `channel_type` 唯一标识。

- 每条消息都属于某个 Channel
- Channel 有订阅者列表，消息投递给所有在线订阅者
- 参见 [[Space多租户]] 了解 Space 场景下的 Channel ID 前缀约定

---

### ChannelType（频道类型）

**定义**：Channel 的类型枚举，决定消息路由语义。

| 值 | 常量 | 含义 |
|----|------|------|
| 0 | ChannelTypeUnknown | 未知（不使用） |
| 1 | ChannelTypePerson | 私聊（DM） |
| 2 | ChannelTypeGroup | 群聊 |
| 3 | ChannelTypeCustomerService | 客服频道 |
| 4 | ChannelTypeCommunity | 社区 |
| 5 | ChannelTypeCommunityTopic | 社区话题 |
| 6 | ChannelTypeInfo | 信息频道 |

---

### Space（空间）

**定义**：Octo 的多租户隔离单元，类似 Slack 的 Workspace。

- 用户加入 Space 后才能使用该空间内的会话和通讯录
- Space 通过 Channel ID 前缀约定实现轻量隔离：`s{spaceId}_{userId/groupId}`
- 数据模型：`space`、`space_member`、`space_preset_channel` 三张表
- API 路径前缀：`/v1/space/`（注意：无 's'，已验证）
- 参见 [[Space多租户]]

---

### BotFather（机器人父亲）

**定义**：Octo 中 AI Bot 的创建与管理模块，设计参考 Telegram BotFather。

- 负责 Bot 的自注册、Token 管理、消息发送、流式输出
- 对话式创建：用户在 Octo 中 @BotFather，发送 `/newbot` 命令创建 Bot
- API 路径前缀：`/v1/bot/`
- 参见 [[Bot系统]]

---

### Robot（机器人模块）

**定义**：dmworkim 中处理 Bot 运行时事件的模块，与 BotFather 互补。

- 负责 Bot 的消息监听、事件投递（通过 gRPC robotMessageListen）、旧版 AppKey 认证
- 数据表：`robot`（Bot 扩展信息）、`robot_menu`（命令菜单）
- API 路径前缀（旧版）：`/v1/robots/:id/:app_key/*`

---

### Bot Token（机器人令牌）

**定义**：BotFather 颁发给 Bot 的身份凭证，用于调用 Bot API。

- **格式**：`bf_` 前缀（如 `bf_abc123xyz`）
- **用途**：`Authorization: Bearer bf_xxx` 请求头，dmworkim 用此验证 Bot 身份
- **替代**：取代旧版 AppKey 认证，更简单，参考 Telegram 设计
- 参见 [[ADR-003-Bot-Token体系]]

---

### API Key（接口密钥）

**定义**：用户或 Bot 的 OpenAI 兼容接口密钥，与 Space 绑定。

- **格式**：`uk_` 前缀
- **用途**：调用 Octo 的 OpenAI 兼容 API 接口（`/v1/chat/completions` 风格）
- **隔离**：每个 Bot 在每个 Space 中有独立的 API Key（`user_api_key` 表绑定 `space_id`）

---

## 消息相关术语

### MessageType（消息类型）

**定义**：WuKongIM 协议层的消息分类。

| 类型值 | 含义 |
|--------|------|
| 1 | 普通文本消息 |
| 2 | 图片消息 |
| 3 | 语音消息 |
| 4 | 视频消息 |
| 5 | 文件消息 |
| 6 | 位置消息 |
| ... | 其他类型 |

---

### ContentType（内容类型）

**定义**：dmwork 业务层的消息内容分类（在 `payload.type` 中传递）。

| 值 | 含义 |
|----|------|
| 1 | 文本（Text） |
| 2 | 图片（Image） |
| 3 | 语音（Voice） |
| 4 | 视频（Video） |
| 5 | 文件（File） |
| 6 | 位置（Location） |
| 7 | 合并转发（MultipleForward） |
| 8 | 卡片（Card/名片） |
| 9 | EmojiReaction |
| 10 | 引用回复（Quote） |
| 11 | 红包（红包消息） |
| 12 | 转账 |
| 13 | 视频通话 |
| 14 | 富文本（RichText） |
| 16 | InviteJoinOrganization（加入组织邀请） |
| 97 | ContentError |
| 99 | 撤回通知 |
| 1000-1022 | 系统消息类型 |
| 1200-1202 | 扩展内容类型 |
| 9989 | CMDMessage（控制指令） |

---

### RECV / RECVACK（接收/接收确认）

**定义**：WuKongIM 二进制协议中的消息接收/确认包对。

- **RECV**（包类型 5）：WuKongIM 推送消息给客户端或 Bot
- **RECVACK**（包类型 6）：客户端/Bot 确认收到，携带 `messageID` + `messageSeq`，防止重复投递

---

### Stream / stream_no（流式消息）

**定义**：AI 逐字输出场景的消息流控制机制。

**流程**：
```
POST /v1/bot/stream/start  → 创建流式会话，返回 stream_no
POST /v1/bot/sendMessage { stream_no: "xxx", ... }  × N次  → 客户端实时更新同一消息气泡
POST /v1/bot/stream/end { stream_no: "xxx" }  → 客户端冻结最终内容
```

- `stream_no`：流式会话标识符，由 WuKongIM 在 `stream/start` 时生成
- 失败时自动降级为一次性 `sendMessage` 发送完整文本

---

### Typing（输入状态）

**定义**：发送方正在输入的状态通知。

- **Bot 端**：`POST /v1/bot/typing { channel_id, channel_type }`
- **客户端**：收到后显示"正在输入..."气泡
- Bot 在开始 AI 推理时发送 typing，每 5 秒续期

---

## OpenClaw 核心术语

### Gateway（网关）

**定义**：OpenClaw 的核心长驻进程，所有 AI 助手的控制面。

- 单端口（默认 18789）同时提供 WebSocket + HTTP
- 负责消息路由、Session 管理、Channel 插件生命周期
- 设计哲学："Gateway is just the control plane"，助手本身才是产品

---

### Agent（代理/助手）

**定义**：OpenClaw 中的 AI 助手实例，有独立的 workspace 和 sessions。

- 每个 Agent 有独立 workspace（SOUL.md、MEMORY.md 等文件）
- 多 Agent 支持：一个 Gateway 可运行多个不同人格的 Agent
- 通过 `bindings` 路由规则将消息分发给特定 Agent

---

### Session（会话）

**定义**：OpenClaw 维护的对话连续性单元，保存对话历史。

- 通过 **Session Key** 唯一标识：`agent:{agentId}:{channel}:{accountId}:{chatType}:{peerId}`
- 示例：`agent:xijing:dmwork:default:direct:user123`
- 每日自动重置（默认凌晨 4 点），或触发 `/new`、`/reset`

---

### Skill（技能）

**定义**：OpenClaw 的功能扩展单元，包含 `SKILL.md` 文件告诉 Agent 如何使用特定工具。

- 内置 53 个技能（apple-reminders、coding-agent、obsidian 等）
- 可通过 ClawHub 技能市场安装第三方技能
- 工作区专属技能优先级最高

---

### Extension（扩展）

**定义**：OpenClaw 的插件包，通过 `openclaw.plugin.json` 声明功能。

- 共 43 个内置扩展，覆盖 20+ IM 渠道、7+ 模型 Provider
- Channel 类扩展（如 `discord`、`feishu`）实现 `ChannelPlugin` 接口
- Provider 类扩展（如 `ollama`）注册新的模型提供商

---

### Provider（提供商）

**定义**：OpenClaw 中 AI 模型的来源，负责将 AI 请求路由到具体模型服务。

- 内置：OpenAI、Anthropic、Google、xAI、Z.AI 等 17+ Provider
- 插件：Ollama、vLLM、SGLang 等本地/自托管 Provider
- 格式：`provider/model`（如 `anthropic/claude-opus-4-6`）

---

### ACP（Agent Collaboration Protocol）

**定义**：OpenClaw 运行外部编码 Agent 的协议（Claude Code、Codex、Pi 等）。

- ACP Session 可绑定到 IM 线程（Discord Thread、Telegram Topic）
- 命令：`/acp spawn codex`、`/acp status`、`/acp cancel`
- 区别于 Sub-agent：ACP 使用外部 harness，Sub-agent 是 OpenClaw 原生

---

### Channel Plugin（渠道插件）

**定义**：OpenClaw 中将 IM 平台接入 Gateway 的标准接口实现。

```typescript
interface ChannelPlugin {
  id: string;
  meta: { id, label, blurb };
  capabilities: { chatTypes, media, reactions, threads };
  outbound: { sendText, sendMedia };
  gateway: { startAccount };  // WebSocket 长连接
}
```

- `openclaw-channel-dmwork` 就是 DMWork 的 Channel Plugin 实现
- 参见 [[愿景与定位]] 了解 OpenClaw + DMWork 的集成关系

## 加密相关术语

### DH（Diffie-Hellman 密钥交换）

**定义**：WuKongIM 连接握手时使用的密钥协商协议，基于 Curve25519 椭圆曲线。

**流程**：
```
客户端生成 Curve25519 密钥对（clientPub, clientPriv）
       ↓
发送 CONNECT 包，携带 clientPub（clientKey 字段）
       ↓
服务端返回 CONNACK，携带 serverPub（serverKey 字段）和 salt
       ↓
双方各自计算：sharedSecret = DH(clientPriv, serverPub)
       ↓
aesKey = MD5(base64(sharedSecret))[0:16]
aesIV  = salt[0:16]
```

- 参见 [[安全与加密]]

---

### AES-128-CBC（加密算法）

**定义**：WuKongIM 中消息 payload 的对称加密算法。

- 密钥（aesKey）和向量（aesIV）通过 DH 密钥交换协商
- 每个连接有独立的密钥，断线重连重新协商
- 加密范围：所有 SEND/RECV 包中的消息 payload

---

### HMAC-SHA256

**定义**：Webhook 签名验证使用的消息认证码算法。

- DMWork 向外部 Webhook 发送事件时，使用 HMAC-SHA256 签名请求体
- 接收方通过验证签名确保消息完整性和来源真实性

---

### Signal Protocol

**定义**：端到端加密消息协议（由 Signal 公司开源）。

- dmworkim 消息表中有 `signal INT DEFAULT 0` 字段标记是否使用 Signal 加密
- iOS 客户端原生支持 Signal 加密（`SignalMessage` 消息类型）
- 提供完美前向保密（PFS）

---

## 技术术语

### EndpointManager（端点管理器）

**定义**：三端客户端（Web/iOS/Android）共用的模块间通信机制。

```
模块A → EndpointManager.invoke("模块B.功能X", params) → 模块B
```

- 类似 iOS URL Scheme / Android Intent，但用于应用内部
- 避免模块间直接依赖，防止循环引用

---

### Snowflake（雪花算法）

**定义**：消息 ID 生成算法，生成全局唯一的 64 位整数 ID。

- `message_id` 字段使用雪花算法生成
- 保证时序性（ID 递增）和分布式唯一性

---

### message_seq（消息序列号）

**定义**：消息在特定频道（Channel）内的递增序列号。

- 每个 Channel 独立计数，用于增量同步
- 客户端通过 `message_seq` 做增量拉取，避免全量同步
